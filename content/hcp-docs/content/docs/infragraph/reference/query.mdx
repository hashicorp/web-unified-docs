# Infragraph query languange reference

The Query language allows users to traverse the graph, filter nodes by properties, and navigate edges between resources.

## Query structure

Infragraph queries are a JSON body with a top level `query` object, which is structured as a nested sequence of alternating `node` and `edge` blocks, defining a traversal path through the graph. The top-level `node` object defines the starting node in the traversal, and the top-level `edge` block finds nodes that match the defined relationship. Each `edge` block contains a `node_query` block, where you can continue to next `node` and `edge` blocks to traverse the graph.

Infragraph supports a maximum traversal depth of 20 nodes.

The following list defines the specification for a `query`:

-	[`query`](#query): map | required
    - [`node`](#node): map | required
        - [`unified_types`](#unified_types): list[string] | required unless [`source_types`](#source-types) defined
        - [`source_types`](#source_types): list[string] | required unless [`unified_types`](#unified-types) defined
        - [`ids`](#ids): list[string] | optional
        - [`alias`](#alias): string | optional
        - [`where`](#where): list[[`match`](#match) and [`filter_group`](#filter_group) objects] | optional
            - [`match`](#match) map | required unless [`filter_group`] defined
                - [`left`](#left): map | required
                    - [`property`](#property): map | required
                        - [`name`](#property): string | required
                        - [`alias`](#property): string | optional
                - [`comparator`](#comparator): string | required
                - [`right`](#right): map | required
                    - [`value`](#right): map | required unless [`value_list`](#right) defined
                    - [`value_list`](#right) map | required unless [`value`](#right) defined
                        - [`values`](#right) list[[`value`](#right)] | required
            - [`filter_group`](#filter_group) map | required unless [`match`](#match) defined
                - [`operator`](#filter_group) string | required
                - [`filter`](#filter_group) list[[`match`](#match)] | required 
    - [`edge`](#edge): map | required
        - [`edge_types`](#edge_types): list[string] | required
        - [`direction`](#direction): string | required
        - [`absent`](#absent): boolean | optional
        - [`node_query`](#node_query): map | required
            - [`node`](#node): map | required
            - [`edge`](#edge): map | optional

## Specification

This section provides details about the fields used to create a `query`.

### query

A `query` block define a single Infragraph query. Each `query` blocks must contain one [`node`](#node) block, and can contain one optional [`edge`](#edge) block.

- Default: none
- This field is required
- Data type: Object containing one or more fields that define a query
    - One required [`node`](#node)
    - One optional [`edge`](#edge)

### node

A `node` block defines which nodes to match at the current step of the traversal. You must specify at least one of `unified_types` or `source_types`, and can optionally provide a specific `ids` filter.

- Default: none
- This field is required
- Data type: Object containing at least one [`unified_types`](#unified_types) or [`source_types`](#source_types). You can define both.
    - [`unified_types`](#unified_types): Matches nodes based on their generalized abstract type across providers. Supports a single `"*"` wildcard per query to match any node type.
    - [`source_types`](#source_types): Matches nodes based on their exact provider-specific type. Does not support wildcards.
    - [`ids`](#ids): Matches specific graph node IDs. Does not support wildcards.
    - [`where`](#where): A list of property filters to apply to the matched nodes.

These dimensions are combined using a logical `AND`. For example, providing both `unified_types=["VIRTUAL_MACHINE"]` and `source_types=["AWS_EC2_INSTANCE"]` looks for nodes that satisfy both constraints.

Example node specification:

```json
{
  "node": {
    "unified_types": ["VIRTUAL_MACHINE"],
    "where": [
      {
        "match": {
          "left": { "property": { "name": "status" } },
          "comparator": "EQUAL",
          "right": { "value": { "string": "running" } }
        }
      }
    ]
  }
}
```

- Default: none
- This field is required unless you define [`source_types`](#source-types). You can define both.
- Data type: List of strings
- Supports a maximum of 5 elements

### ids

Matches specific graph node IDs. Does not support wildcards.

- Default: none 
- Data type: List of strings
- Supports a maximum of 10 elements

### alias

Optional reference alias to assign to this node. You can reference this alias in [`left`](#left) or [`right`](#right) blocks in a [`where`](#where) clause.

- Default: none
- Data type: string

### where

A list of property filters to apply to the matched nodes. Infragraph matches top-level entries in a `where` block with a logical `AND` operator. Use a `filter_group` to match multiple filters with an `OR` operator. 

You can combine multiple `filter_group` blocks to create a combination of `AND` and `OR` operators. For example, to match the logic of `(A OR B) AND (C OR D)` you can create one `filter_block` to match at least one `A` or `B` filter, and it must also match one of the `C` or `D` filters.

- Default: none
- Data type: A list of [`match`](#match) and [`filter_group`](#filter_group) objects
- Supports a maximum of 10 elements

### match

Finds a `node` object that satisfies every `match` clause or the `filter_group` clause.

- Default: none
- Data type: map
    - [`left`](#left): The property to inspect.
    - [`comparator`](#comparator): The operation to perform.
    - [`right`](#right): The value to compare against the `left` value. 

The following is am example `match` clause using the `IN` comparator:

```json
{
  "match": {
    "left": { "property": { "name": "region" } },
    "comparator": "IN",
    "right": {
      "value_list": {
        "values": [
          { "string": "us-east-1" },
          { "string": "us-west-2" }
        ]
      }
    }
  }
}
```

### left

The property to inspect in a `match` clause. 

- Default: none
- Data type: One `property` object

#### property

Each `property` object contains the following fields:

| Field | Description | Required | 
| ----- | ----------- | -------- |
| `name` | The name of the property to compare to the value defined in the `right` field | Yes |
| `alias` | Identifies a node elsewhere in the traversal so that the filter compares against that node's property rater than the current node | No |

### comparator

The operation to perform to compare the values of the `left` and `right` fields of a `match` clause.

- Default: none
- Data type: string

Refer to [Comparators](#comparators) for more information on the supported comparators.

### right 

The value to compare against the `left` value. 

- Default: none
- Data type: One of `property`, `value`, or `value_list`

Refer to [`property`](#property) for information on the `property` field.

A `value` is an object that contains exactly one data type and one value. The following is a list of example `value` objects:

```json
"right": { "value": { "string": "us-east-1" } }
"right": { "value": { "int": 42 } }
"right": { "value": { "float": 1.5 } }
"right": { "value": { "bool": true } }
"right": { "value": { "timestamp": "2025-02-07T23:00:00-01:00" } }
```

A `value_list` is a list of `value` objects. Infragraph only supports `value_list` objects when you set the `comparator` to `IN` or `NOT_IN`. This block supports a maximum of 10 elements, and all elements must be of the same type.


The following is a an example `right` object that provides a `value_list`:

```json
"right": {
  "value_list": {
    "values": [
      { "string": "us-east-1" },
      { "string": "us-west-2" }
    ]
  }
}
```

### filter_group

Defines one `operator` one or more `filter` used to narrow down the results of the traversal.

- Default: none
- Data type: Object containing one `operator` and one `filter`.
    - One required `operator`. Must be a string with the value `AND` or `OR`.
    - One required `filter`. Must be a list of [`match`](#match) blocks.
- Supports a maximum of 10 `filter` blocks

The following examples defines a `filter_group` with two filters. Since the `operator` is set to `OR`, this filter group returns nodes that match either filter: 

```json
{
  "filter_group": {
    "operator": "OR",
    "filter": [
      {
        "match": {
          "left": { "property": { "name": "environment" } },
          "comparator": "EQUAL",
          "right": { "value": { "string": "production" } }
        }
      },
      {
        "match": {
          "left": { "property": { "name": "is_critical" } },
          "comparator": "EQUAL",
          "right": { "value": { "bool": true } }
        }
      }
    ]
  }
}
```


### edge

An `edge` block dictates how to traverse from the currently matched node to the next set of nodes.

- Default: none
- Data type: Object
    - One required [`edge_types`](#direction). Supports a single `*` wildcard per query for any edge type.
    - One required [`direction`](#direction).
    - One optional [`absent`](#absent).
    - One required [`node_query`](#node_query).

The following is an example `edge` block:

```json
{
  "edge": {
    "edge_types": ["ATTACHES_TO"],
    "direction": "OUTGOING",
    "node_query": {
      "node": {
        "unified_types": ["ROLE"]
      }
    }
  }
}
```

### edge_types

The relationships to traverse, such as `["MANAGES", "CONTAINS"]`. Supports a single `*` wildcard per query for any edge type.

- Default: none
- This field is required
- Data type: List of strings
- Supports a maximum of 5 elements

### direction

The direction of the relationship to traverse.

- Default: `OUTGOING`
- This field is required

The following is a list of valid values for this field:

- `OUTGOING`
- `INCOMING`
- `BOTH`

### absent

If set to `true`, the query behaves as a `NOT` step, matching nodes that do not have the specified edge.

- Default: none
- Data type: boolean

### node_query

The nested next step in the traversal, containing another `node` and optional `edge`.

- Default: none
- Data type: Object containing one [`node`](#node) object.

## Comparators

Infragraph supports the following comparators:


| Comparator              | Right operand                  | Allowed value types                              | Notes                                                                                    |
|-------------------------|--------------------------------|--------------------------------------------------|------------------------------------------------------------------------------------------|
| `EQUAL`                 | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Exact equality.                                                                          |
| `NOT_EQUAL`             | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Inverse of `EQUAL`.                                                                      |
| `GREATER_THAN`          | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Ordered comparison.                                                                      |
| `LESS_THAN`             | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Ordered comparison.                                                                      |
| `GREATER_THAN_OR_EQUAL` | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Ordered comparison.                                                                      |
| `LESS_THAN_OR_EQUAL`    | `value` or `property`          | `string`, `int`, `float`, `bool`, `timestamp`    | Ordered comparison.                                                                      |
| `EXISTS`                | none                           | n/a                                              | Right operand must be omitted entirely.                                                  |
| `DOES_NOT_EXIST`        | none                           | n/a                                              | Right operand must be omitted entirely.                                                  |
| `IS_NOT_EMPTY`          | none                           | n/a                                              | Right operand must be omitted entirely.                                                  |
| `IN`                    | `value_list` (required)        | `string`, `int`, `float` (homogeneous list)      | `bool` and `timestamp` values are rejected inside `value_list`.                          |
| `NOT_IN`                | `value_list` (required)        | `string`, `int`, `float` (homogeneous list)      | Same restrictions as `IN`.                                                               |
| `STARTS_WITH`           | `value` (required)             | `string` only, non-empty                         | Right value must be a non-empty `string` ≤ 1024 chars; any other type/shape is rejected. |

## Example query

The query below is an example of how to nest `node` and `edge` blocks to traverse Infragraph's data. This example finds all `VIRTUAL_MACHINE` nodes that run a `VIRTUAL_MACHINE_IMAGE` that HCP Packer did not create.

```json
{ 
  "node": { 
    "unifiedTypes": ["VIRTUAL_MACHINE"]
  }, 
  "edge": { 
    "edgeTypes": ["RUNS"],
    "direction": "EDGE_DIRECTION_OUTGOING",
    "nodeQuery": { 
      "node": { 
        "unifiedTypes": ["VIRTUAL_MACHINE_IMAGE"]
      }, 
      "edge": { 
        "edgeTypes": ["PRODUCES"],
        "direction": "EDGE_DIRECTION_INCOMING",
        "absent": true, 
        "nodeQuery": { 
          "node": { 
            "unifiedTypes": ["PACKER_ARTIFACT"]
          } 
        } 
      } 
    } 
  } 
}
```

You can continue to nest `node` and `edge` blocks to refine your query.

Refer to [Example queries](/hcp/docs/infragraph/query/examples) for more example queries.

## Limitations

The Query API enforces the following structural limits on queries. Exceeding these limits will result in a `400 Bad Request` response containing detailed field violations.

- **Traversal Depth:** Maximum 20 nested `node`/`edge` pairs.
- **Node Filters:**
  - `unified_types`: Max 5 elements.
  - `source_types`: Max 5 elements.
  - `ids`: Max 10 elements.
  - Only 1 wildcard (`*`) is allowed across all node types in the entire query.
- **Edge Filters:**
  - `edge_types`: Max 5 elements.
  - Only 1 wildcard (`*`) is allowed across all edge types in the entire query.
- **Predicates (`where` clauses):**
  - Max 10 items in a single `where` list.
  - Max 10 items inside a single `filter_group`.
  - Max predicate nesting depth of 10.
- **Values:**
  - String values in filters are capped at 1024 characters.
  - `value_list` (for `IN`/`NOT_IN`) is limited to a maximum of 10 values, and all values must be of the exact same type.
- **Property Mask:**
  - The `node_properties` string cannot exceed 2048 characters.
