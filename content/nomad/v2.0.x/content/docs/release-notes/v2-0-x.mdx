## 2.0.4 release highlights

Release Date:  7 July 2026

### Optional `Init` and `Shutdown` functions for task driver plugins

Task driver plugins now support optional `Init` and `Shutdown` functions.
Nomad calls `Init` after `SetConfig` to allow the driver to perform setup
before receiving requests. When Nomad shuts down the driver, it calls
`Shutdown` to allow cleanup before the driver process is terminated.

Refer to the [task driver plugin API
documentation](/nomad/plugins/author/task-driver) for details.

### Consul token fallback for workloads without workload identity

Allocations without workload identity configured for Consul fall back to using
the Nomad client agent's own Consul token for `service` blocks, `template`
blocks, and Consul service mesh.

For `template` blocks, set `use_client_consul_token = true` in the [client
`template` block
configuration](/nomad/docs/configuration/client#use_client_consul_token) to
allow task templates to use the Nomad client agent's Consul token when workload
identity is not configured.

Refer to [Consul without workload
identity](/nomad/docs/secure/acl/consul#consul-without-workload-identity) for
details.

### New `run_on_first_render` parameter for `change_script` block

The `run_on_first_render` parameter in the `change_script` block controls
whether the configured script executes after the initial template render in
addition to executing on subsequent template changes. Set
`run_on_first_render = true` to run the script after the initial render.

Refer to the [`change_script` block
documentation](/nomad/docs/job-specification/change_script#run_on_first_render)
for details.

### Server join deprecations

Using the `server join` command or the [Join Agent
API](/nomad/api-docs/agent#join-agent) without authentication is deprecated.
Nomad 2.1.0 will require using a token with the `agent:write` permission.

We are removing the following server configuration parameters in Nomad 2.1.0:

- `server.retry_join`
- `server.retry_interval`
- `server.retry_max`
- `server.start_join`

Use [`server.server_join`](/nomad/docs/configuration/server#server_join)
instead.

Refer to the [Nomad 2.0.4 upgrade-specific
documentation](/nomad/docs/upgrade/upgrade-specific#nomad-204) for details.

### Docker task driver `allowed_modes` configuration parameter

The `allowed_modes` configuration parameter enables allowlists for the `pid`,
`ipc`,`userns`, and `uts` container namespaces. Refer to the [Docker task driver
`allowed_modes` parameter
documentation](/nomad/docs/deploy/task-driver/docker#allowed_modes) for details.

## 2.0.3 release highlights

Release Date: 9 June 2026

### New `max_run_duration` field added to job specification group

For batch and system batch jobs, `max_run_duration` defines the
maximum run time for allocations in the group. Refer to [the `max_run_duration`
parameter documentation](/nomad/docs/job-specification/group#max_run_duration) for details.

Additionally, we added the
`nomad.client.allocs.max_run_duration.configured_seconds` and
`nomad.client.allocs.max_run_duration.remaining_seconds` metrics. Refer to [the
allocation metrics reference
documentation](/nomad/docs/reference/metrics#allocation-metrics) for more
information.

### Add custom tokens with `nomad acl token create` command

The `nomad acl token create` command now supports a `-token` flag to specify a
custom token value when creating an ACL token. Refer to [the `nomad acl token
create` command reference](/nomad/commands/acl/token/create#usage) for more
details.

### Mark nodes as ineligible at client initialization

Use the `default_ineligible` client configuration parameter to mark nodes as
ineligible at client initialization. Refer to [the `default_ineligible`
parameter documentation](/nomad/docs/configuration/client#default_ineligible)
for more information.

### Trigger a reload of the agent's configuration with the API

The `PUT /v1/agent/reload` API endpoint triggers a reload of the agent's
configuration. Refer to [the Agent reload API endpoint reference
documentation](/nomad/api-docs/agent#reload-agent) for more details.

### Workload identity extra claims

You may add extra claims to workload identities based on the namespace
configuration. Refer to the following for details:

- [Workload identity extra claims documentation](/nomad/docs/concepts/workload-identity#workload-identity-extra-claims)
- [Namespace `required_extra_claims` parameter documentation](/nomad/docs/other-specifications/namespace#required_extra_claims)
- [Namespace `optional_extra_claims` parameter documentation](/nomad/docs/other-specifications/namespace#optional_extra_claims)

## 2.0.1 release highlights

Release Date: 12 May 2026

### Critical security updates

This release includes two critical security updates:
- Logmon setup now prevents a task from swapping out the logmon FIFOs for
  symlinks to read arbitrary files on the host. This vulnerability is
  CVE-2026-6959.
- Dynamic Host Volume creation now validates that the plugin ID is a valid
  plugin in all code paths, preventing an authorized user from running arbitrary
  executables on the host as though they were plugins. This vulnerability is
  CVE-2026-7474.

### Breaking changes

Task drivers that support filesystem isolation such as `docker` now have their
allocation log directories bind mounted read-only.

### Outbound RPC connection timeout parameter

The [`dial_timeout` agent configuration
parameter](/nomad/docs/configuration/rpc) controls the maximum amount of time
allowed to establish new outbound RPC connections to peer agents.

### Node limit scheduler parameter

To increase placement randomization and reduce scheduler contention, set the
`NodeLimitForFeasibilityChecks` scheduler configuration option. Refer to the
following documentation for more details:

- [Job scheduling
  performance](/nomad/docs/job-scheduling#scheduling-performance)
- Job specification affinity and spread configuration parameters
  - [`affinity` block](/nomad/docs/job-specification/affinity)
  - [`spread` block](/nomad/docs/job-specification/spread)
- [Scheduler Operator HTTP API `NodeLimitForFeasibilityChecks`
  parameter](/nomad/api-docs/operator/scheduler#nodelimitforfeasibilitychecks)
- The [`nomad operator scheduler get-config` command
  reference](/nomad/commands/operator/scheduler/get-config)
- The [`nomad operator scheduler set-config` command
  reference](/nomad/commands/operator/scheduler/set-config)

### Warning on job service without shutdown delay configured

Nomad emits a warning when a job is submitted with services defined but no service at
the task level has `shutdown_delay` set.

## 2.0.0 release highlights

Release Date: 21 April 2026

### Command to migrate Raft log store to Write-Ahead Log (WAL) backend

The `nomad operator raft migrate-backend` command migrates the Raft log store
from the BoltDB backend to the Write-Ahead Log (WAL) backend. Compared to
BoltDB, the WAL backend provides better performance characteristics for
write-heavy workloads. However, once migrated to WAL, reverting back to BoltDB
requires restoring from a snapshot taken before the migration.

We added the following:

- [`nomad operator raft migrate-backend` command reference](/nomad/commands/operator/raft/migrate-backend)
- Raft log store details to the [`/v1/agent/self` API
  endpoint](/nomad/api-docs/agent#query-self) response
- [`raft_logstore` parameter](/nomad/docs/configuration/server#raft_logstore) to
  server configuration
- [WAL backend metrics](/nomad/docs/reference/metrics#wal-backend-metrics) to the Raft log store metrics

We deprecated the server configuration's [`raft_boltdb`
parameter](/nomad/docs/configuration/server#raft_boltdb).

### License and configuration changes to enable use of IBM Passport Advantage Online (PAO) <EnterpriseAlert inline/>

We extended Nomad's existing licensing and reporting functionality to support
parsing and reporting with the IBM PAO license.
