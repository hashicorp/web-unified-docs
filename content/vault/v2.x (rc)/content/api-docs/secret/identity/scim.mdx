# SCIM API

@include 'alerts/enterprise-only.mdx'

@include 'alerts/beta.mdx'

Use the SCIM API to provision Vault identity entities and internal groups from
external identity systems.

Before you call these endpoints, activate the SCIM feature flag `enable-scim`
through the [activation flags API](/vault/api-docs/system/activation-flags).

## Authentication and Authorization

SCIM requests are authenticated via a Vault token like usual. However, for a SCIM
request to be authorized, beyond clearing the usual ACL policy evaluation, the
entity ID associated with the token match be the `access_grant_principal` field of
a SCIM client in the same namespace. That's also how Vault resolves the identity of
the calling SCIM client.

## SCIM client scope

- SCIM clients are namespace-scoped.
- SCIM users and groups are only visible to the SCIM client that created them.
- Only the SCIM client is able to manage the lifecycle of resources it created.
- Only the SCIM client can set SCIM-related fields of resources it created.
- Vault returns SCIM protocol responses with the `application/scim+json`
  content type.

## SCIM client endpoints

Use SCIM client endpoints to define the external provisioning system that can
call the SCIM protocol endpoints.

| Method | Path |
| :----- | :--- |
| `POST` | `/identity/scim/client/:client_name` |
| `GET` | `/identity/scim/client/:client_name` |
| `DELETE` | `/identity/scim/client/:client_name` |
| `LIST` | `/identity/scim/client` |

### Parameters

- `client_name` `(string: <required>)` - Name of the SCIM client. The name must
  be unique within the namespace.
- `access_grant_principal` `(string: <required>)` - ID of an existing Vault
  entity in the same namespace. Vault uses this entity to authorize SCIM
  protocol requests.
- `alias_mount_accessor` `(string: <optional>)` - Auth mount accessor used to
  create an entity alias for each SCIM user. The accessor must refer to an
  existing non-local auth mount in the same namespace.

~> **Note:** Vault does not let you change or clear `alias_mount_accessor`
after you create the SCIM client.

~> **Note:** Deleting a SCIM client will mark it as `deleting: true` and launch
an asynchronous removal of all entities, aliases and groups owned by that client.

### Sample payload

```json
{
  "access_grant_principal": "d8be46f8-b754-7d85-2dc6-37e15a24d7d5",
  "alias_mount_accessor": "auth_userpass_7d1d6f49"
}
```

## User endpoints

Vault supports the following standard SCIM `/Users` endpoints:

| Method | Path |
| :----- | :--- |
| `POST` | `/identity/scim/v2/Users` |
| `GET` | `/identity/scim/v2/Users` |
| `GET` | `/identity/scim/v2/Users/:id` |
| `PATCH` | `/identity/scim/v2/Users/:id` |
| `PUT` | `/identity/scim/v2/Users/:id` |
| `DELETE` | `/identity/scim/v2/Users/:id` |

SCIM users map to Vault entities.

- `id` maps to the entity ID.
- `userName` maps to the entity name.
- `externalId` maps to the entity external ID.
- `active` maps to the inverse of the entity `disabled` state.

Vault requires `userName` and `externalId` on create. Vault treats
`externalId` as immutable after creation.

### Supported filters

Vault supports simple `eq` filters on `/Users` for the following fields:

- `userName`
- `externalId`
- `active`

Vault rejects complex filter expressions and other filter operators.

### Supported PATCH behavior

Vault only supports a single `replace` operation for the `active` field when
you patch a user.

### Sample create user payload

```json
{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:User",
    "urn:ietf:params:scim:schemas:extension:vault:2.0:Metadata"
  ],
  "userName": "jane.doe",
  "externalId": "00u123456789",
  "active": true,
  "urn:ietf:params:scim:schemas:extension:vault:2.0:Metadata": {
    "items": [
      {
        "key": "department",
        "value": "engineering"
      }
    ]
  }
}
```

## Group endpoints

Vault supports the following standard SCIM `/Groups` endpoints:

| Method | Path |
| :----- | :--- |
| `POST` | `/identity/scim/v2/Groups` |
| `GET` | `/identity/scim/v2/Groups` |
| `GET` | `/identity/scim/v2/Groups/:id` |
| `PATCH` | `/identity/scim/v2/Groups/:id` |
| `PUT` | `/identity/scim/v2/Groups/:id` |
| `DELETE` | `/identity/scim/v2/Groups/:id` |

SCIM groups map to Vault internal identity groups.

- `id` maps to the group ID.
- `displayName` maps to the group name.
- `members[].value` must contain Vault user entity IDs.
- Vault only allows members owned by the same SCIM client.
- Groups cannot be added as members to other groups.

### Supported PATCH behavior

Vault supports group `PATCH` for member add and remove operations.

- Use the `members` path or omit the path.
- Vault rejects other group patch operations.

### Sample create group payload

```json
{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:Group"
  ],
  "displayName": "vault-admins",
  "members": [
    {
      "value": "2cc3c936-fb4e-4635-b213-9cf55b87f2d4"
    }
  ]
}
```

## Discovery endpoints

Vault supports the following standard SCIM discovery endpoints:

| Method | Path |
| :----- | :--- |
| `GET` | `/identity/scim/v2/ResourceTypes` |
| `GET` | `/identity/scim/v2/ResourceTypes/:id` |
| `GET` | `/identity/scim/v2/Schemas` |
| `GET` | `/identity/scim/v2/Schemas/:id` |
| `GET` | `/identity/scim/v2/ServiceProviderConfig` |

`/identity/scim/v2/ServiceProviderConfig` is available without authentication.

Vault also declares `/identity/scim/v2/Me`, but the endpoint returns
`501 Not Implemented`.

## Limitations

Many of these will be covered in upcoming Vault releases:

- Activating the SCIM feature flag is a one-way action.
- Vault does not support `/Me`.
- Vault does not support SCIM bulk operations.
- Vault does not support sort, ETag, or change password operations.
- User `PATCH` only supports a single `replace` operation for `active`.
- Group `PATCH` only supports member add and remove operations.
- Vault supports simple `eq` filters on `/Users` for `userName`, `externalId`,
  and `active`.
- Vault does not support group filters or complex SCIM filter expressions.
- Group members must use Vault user entity IDs, not `userName` or
  `externalId`.
- `externalId` is required when you create a user and you cannot update it
  later.
- `alias_mount_accessor` is immutable after client creation and remains tied to
  the auth mount strategy you choose for that client.
- `alias_mount_accessor` cannot refer to a local mount, only non-local ones.
- Making existing resources become managed by a SCIM client is not supported.
- Migrating resources between SCIM clients is not supported.
- Making SCIM-managed resources unmanaged is not supported.
