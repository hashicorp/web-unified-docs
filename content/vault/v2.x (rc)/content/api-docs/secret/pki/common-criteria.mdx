~> **Note**: This feature flag is being introduced before starting NIAP
   certification in order to address certain compatibility differences between
   Vault behaviour and the Common Criteria Requirements.  Vault PKI does not yet
   have NIAP certification, and in the future, enabling this feature flag will be
   necessary but not sufficient to run Vault in a manner compliant with common
   criteria.

# Common Criteria Flag

The feature flag is "common_criteria_mode", it's set to "true" (the string) in
Vault configuration to enable.

## TLS Versions

Common Criteria limits the list of allowed TLS versions to forbid `TLS_RSA_WITH_AES_128_GCM_SHA256` and `tls.TLS_RSA_WITH_AES_256_GCM_SHA384`

## Not_Before

When Common Criteria mode is enabled, the not_before field on the following
endpoints is ignored, and instead functions as if set to `0`:

| Method | Path |
| :----- | :--- |
| `POST` | `/pki/root/sign-intermediate/` |
| `POST` | `/pki/issuer/:issuer_ref/sign-intermediate/` |
| `POST` | `/pki/sign-verbatim(/:name)` |
| `POST` | `/pki/issuer/:issuer_ref/sign-verbatim(/:name)` |
| `POST` | `/pki/roles/:name` |
| `PATCH` | `/pki/roles/:name` |

This behaviour may cause problems in environments with long delays and clock
skew, especially backwards compatibility with older systems that are not accounting
for leap seconds.  In those cases, certificates being issued might not immediately
be seen as valid by systems using a different clock.

## Issuance Requires Full Chain of Trust

Certificates issued by a CA (Certificate Authority, or issuer) by Vault PKI secrets
engine are validated against the chain configured on that Certificate Authority
(or issuer).  In typical operation, it is valid for that chain not to include a
root certificate, and no warning nor error is thrown.  Vault instead assumes this
behavior is intentional, and that an intermediate issuer is trusted by the system
downstream.

When running in Common Criteria mode, it is not possible to issue a certificate if
there is not a complete chain of trust from the issued certificate up to the root
certificate authority.  This will instead throw an error.  Attempts to read or
modify an issuer which does not have a complete chain of trust will warn the user.

### Validated Upload

Since it is not possible to use a Certificate Authority which does not have a
complete chain of trust when operating in Common Criteria mode, in this mode the
Vault PKI Engine will reject import of certificate authorities without that chain
of trust.

To import a issuer (a Certificate Authority whose key material is managed by Vault
PKI), it is necessary to first (or simultaneously) import the Certificate
Authorities which compose the chain of trust of that issuer.

For instance the following sequence of steps will error:
1. Generate a CSR inside Vault
2. Sign that CSR by an external CA
3. Import just the signed CSR (certificate) into Vault

But this sequence of steps will succeed:
1. Generate a CSR inside Vault
2. Sign that CSR by an external CA
3. Import that CA and the signed CSR into Vault

## Extended Key Usage Validation

In typical usage, Vault does not check that certificates specifying Extended
Key Usages (EKUs) have a key usage required to make that Extended Key Usage make
sense.  It is rare for this to be checked by external systems, and mistakes with
Key Usages are common.  When running under Common Criteria mode, these dependency
key usages are now enforced.  See [RFC 5280 Section 4.2.1.12 - Extended Key
Usage](https://datatracker.ietf.org/doc/html/rfc5280#section-4.2.1.12) for more
information.
