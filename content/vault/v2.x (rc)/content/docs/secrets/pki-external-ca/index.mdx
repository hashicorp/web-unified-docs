# PKI external CA secrets engine

<EnterpriseAlert product="vault" />

The PKI External CA secrets engine is a HashiCorp Enterprise plugin that enables Vault to acquire signed leaf certificates from external Certificate Authorities (CAs) through the ACME (Automatic Certificate Management Environment) protocol. Organizations can leverage public CAs like Let's Encrypt, DigiCert, or other ACME-compliant CAs while maintaining Vault's centralized certificate management capabilities.

## Overview

The PKI External CA secrets engine bridges the gap between Vault's certificate management capabilities and external ACME-compliant Certificate Authorities. Instead of Vault acting as its own CA (as with the standard PKI secrets engine), this plugin allows Vault to:

1. **Manage ACME accounts** with external CAs
2. **Automate certificate acquisition** through the ACME protocol
3. **Handle challenge fulfillment** for domain validation
4. **Cache and distribute certificates** to applications
5. **Manage certificate lifecycle** including renewal and revocation

This is particularly useful for organizations that need to use publicly trusted certificates from established CAs while maintaining Vault's security model and API.

## Setup

### Enable the secrets engine

```shell-session
$ vault secrets enable -path=pki-external-ca vault-plugin-secrets-pki-external-ca
```

### Configure an ACME account

#### For Let's Encrypt (production)

```shell-session
$ vault write pki-external-ca/config/acme-account/letsencrypt-prod \
    directory_url="https://acme-v02.api.letsencrypt.org/directory" \
    email_contacts="admin@example.com,security@example.com" \
    key_type="ec-256"
```

#### For Let's Encrypt (staging)

```shell-session
$ vault write pki-external-ca/config/acme-account/letsencrypt-staging \
    directory_url="https://acme-staging-v02.api.letsencrypt.org/directory" \
    email_contacts="admin@example.com" \
    key_type="ec-256"
```

#### For CAs requiring External Account Binding (EAB)

Some ACME servers require External Account Binding for account registration:

```shell-session
$ vault write pki-external-ca/config/acme-account/digicert \
    directory_url="https://acme.digicert.com/v2/acme/directory" \
    email_contacts="admin@example.com" \
    eab_kid="your-eab-key-id" \
    eab_key="base64-encoded-eab-key" \
    key_type="rsa-2048"
```

### Create roles

Roles define which domains can be requested and how certificates are issued:

```shell-session
# Simple role for specific domains
$ vault write pki-external-ca/role/web-servers \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="example.com,www.example.com,api.example.com" \
    allowed_domain_options="bare_domains"

# Role with wildcard support
$ vault write pki-external-ca/role/wildcard-certs \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="*.example.com" \
    allowed_domain_options="wildcards" \
    allowed_challenge_types="dns-01"

# Role with subdomain support
$ vault write pki-external-ca/role/subdomains \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="example.com" \
    allowed_domain_options="bare_domains,subdomains"

# Role with identity templating
$ vault write pki-external-ca/role/user-certs \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="{{identity.entity.name}}.users.example.com" \
    allowed_domain_options="bare_domains"
```

## Certificate workflows

The PKI External CA secrets engine supports two distinct workflows for requesting certificates,
similar to the [sign](/vault/api-docs/secret/pki#sign-certificate) and
[issue](/vault/api-docs/secret/pki#generate-certificate-and-key) endpoints in the standard PKI
secrets engine. 

You can choose the workflow that best fits your security requirements and operational needs.

### Workflow comparison

| Feature | CSR Workflow | Identifier Workflow |
|---------|-------------|---------------------|
| **Private key location** | Client generates and keeps | Vault generates and provides |
| **Key transmission** | Never leaves client | Vault transmits to client after generation |
| **Security model** | Higher - key never transmitted | Standard - Vault transmits key once |
| **Complexity** | Higher - client must generate CSR and associated key | Lower - client only provides identifiers |
| **Certificate caching** | Not supported | Vault caches both certificate and private key |

### When to use each workflow

**Use the CSR workflow when:**
- You require maximum security and want private keys to never leave the client system
- You have existing key management processes and infrastructure
- You need to comply with security policies that prohibit key transmission
- You deploy single-instance applications where key sharing is not needed

**Use the identifier workflow when:**
- You prioritize operational simplicity over maximum security
- You want Vault to handle key generation and management
- You need to retrieve both the certificate and key at a later time
- Your security model allows for encrypted key transmission over TLS

## Certificate workflows

### Identifier workflow

In the identifier workflow, you provide domain names and Vault generates the private key. The private key
and certificate will be cached by Vault and can be retrieved later.

```shell-session
# Create order
$ vault write pki-external-ca/role/web-servers/new-order \
    identifiers="www.example.com,api.example.com"

# Response includes order_id
# order_id: 01936d8e-7c3a-7890-b123-456789abcdef

# Check order status and get challenges
$ vault read pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef/challenge

# Fulfill challenges (place HTTP tokens or DNS records)
# ...

# Notify Vault that challenges are ready
$ vault write pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef/fulfilled-challenge \
    identifier="www.example.com" \
    challenge_type="http-01"

# Wait for certificate issuance (automatic)
# Check status
$ vault read pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef

# Retrieve certificate when ready
$ vault read pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef/fetch-cert
```

### CSR workflow

In the CSR workflow, you generate your own private key and CSR retrieving back only the signed certificate back from Vault:

```shell-session
# Generate private key and CSR locally
$ openssl req -new -newkey rsa:2048 -nodes \
    -keyout server.key \
    -out server.csr \
    -subj "/CN=www.example.com"

# Create order with CSR
$ vault write pki-external-ca/role/web-servers/new-order \
    csr=@server.csr

# Follow same challenge fulfillment process as identifier workflow
# ...

# Retrieve certificate (without private key since you provided it)
$ vault read pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef/fetch-cert
```

## Role configuration options

Within each role, you can specify which ACME account to use, which domains are allowed,
and how challenges should be fulfilled.

### Limiting allowed domains

Roles support several options for controlling which domains can be requested. Each option
influences how Vault matches the allowed_domains against the identifiers in the new-order request.

#### Bare domains

Allow matching domains without any subdomain prefix, such as example.com:

```shell-session
$ vault write pki-external-ca/role/example \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="example.com,www.example.com" \
    allowed_domain_options="bare_domains"

# Allows: example.com, www.example.com
# Denies: api.example.com, sub.example.com
```

#### Subdomains

Allow subdomains of specified domains:

```shell-session
$ vault write pki-external-ca/role/example \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="example.com" \
    allowed_domain_options="subdomains"

# Allows: www.example.com, api.example.com, sub.api.example.com
# Denies: example.com, example.org
```

#### Wildcards

Allow wildcard certificates:

```shell-session
$ vault write pki-external-ca/role/example \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="*.example.com" \
    allowed_domain_options="wildcards" \
    allowed_challenge_types="dns-01"

# Allows: *.example.com
# Denies: example.com, *.*.example.com
# Note: Wildcard certificates require DNS-01 challenge
```

#### Globs

Allow glob patterns in domain specifications:

```shell-session
$ vault write pki-external-ca/role/example \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="*.prod.example.com,*.staging.example.com" \
    allowed_domain_options="globs"

# Allows: api.prod.example.com, web.staging.example.com
# Denies: example.com, prod.example.com
```

#### Identity templating

Roles can use ACL path templating to create dynamic domain allowlists based on the authenticated entity:

```shell-session
$ vault write pki-external-ca/role/user-certs \
    acme_account_name="letsencrypt-prod" \
    allowed_domains="{{identity.entity.name}}.users.example.com" \
    allowed_domain_options="bare_domains"
```

When user "alice" requests a certificate:
- Allowed: `alice.users.example.com`
- Denied: `bob.users.example.com`

Available template variables:
- `{{identity.entity.id}}` - Entity ID
- `{{identity.entity.name}}` - Entity name
- `{{identity.entity.metadata.<key>}}` - Entity metadata
- `{{identity.groups.names.<group>}}` - Group membership

## Monitoring and troubleshooting

### Check order status

```shell-session
$ vault read pki-external-ca/role/web-servers/order/01936d8e-7c3a-7890-b123-456789abcdef/status
```

Order statuses:
- `new` - Order created, not yet submitted to ACME server
- `submitted` - Submitted to ACME server
- `awaiting_challenge_fulfillment` - Waiting for client to fulfill challenges
- `notify_acme_server_challenges_completed` - Ready to notify ACME server
- `processing_challenge` - ACME server is validating challenges
- `fetching_certificate` - Retrieving issued certificate
- `completed` - Certificate issued successfully
- `expired` - Order expired before completion
- `revoked` - Vault revoked the certificate
- `error` - An error occurred

### List active orders

```shell-session
$ vault list pki-external-ca/role/web-servers/active-orders
```

### Lookup certificate by serial

```shell-session
$ vault read pki-external-ca/lookup/cert/03:e7:1f:a2:3d
```

### View ACME account details

```shell-session
$ vault read pki-external-ca/config/acme-account/letsencrypt-prod
```

## Known limitations

### Load-balanced applications with multiple instances

The current implementation works best when a single orchestrator or instance manages certificate requests.
If you deploy multiple instances of an application behind a load balancer, where each instance independently requests
certificates from Vault, you may encounter challenges:

**The problem:**
- Multiple instances may create separate orders for the same identifiers simultaneously
- Each order generates unique challenge tokens specific to that order
- The ACME CA may validate challenges against a different instance than the one that received the challenge information
- This can cause validation failures and certificate issuance problems

**Recommended workarounds:**

1. **Single orchestrator pattern**: Designate one instance or orchestrator to handle certificate requests. After Vault issues the certificate, distribute it to all instances that need it.

2. **Identifier workflow with caching**: Use the identifier workflow (not CSR workflow) so Vault generates and caches both the certificate and private key. Vault then allows all instances to retrieve the same certificate and key.

### IP address identifiers

The plugin currently does not support requesting certificates with IP address identifiers. 

### ACME Renewal Information (ARI)

The plugin does not currently support the ACME Renewal Information (ARI) extension ([RFC 9773](https://datatracker.ietf.org/doc/rfc9773/)). 
This extension allows ACME servers to notify clients when they should renew certificates, rather than clients deciding renewal timing independently.

### Automatic ACME account key rotation

Automatic rotation of ACME account keys based on key age. 

Until this feature is available, you can manually rotate account keys using the `/config/acme-account/:name/rotate-key` endpoint.
