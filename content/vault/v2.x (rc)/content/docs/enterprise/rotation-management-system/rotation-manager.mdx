# Rotation manager

@include 'alerts/enterprise-and-hcp.mdx'

Vault Enterprise includes rotation manager, a system that runs in the background to
automate schedule-based or period-based credential rotation. The rotation manager maintains
a priority queue of credentials, checks for pending rotations every 5 seconds,
and dispatches rotation requests to the appropriate plugin. All rotation states
persist to storage for recovery after restarts or failovers.

The rotation manager marks a rotation entry as orphaned when it
exceeds the maximum retry limits defined in a rotation policy. Orphaning halts
the rotation attempts and requires manual intervention to resolve the
error. Refer to the [rotation orphan documentation](/vault/api-docs/system/rotation-orphans)
for more information.

## High availability

The rotation manager runs only on the active node of a cluster. When leadership
changes, the active node restores rotation state from storage and resumes
processing the queue.

For replicated clusters:

- **Shared mounts**: Rotations occur on the primary cluster's active node.
- **Local mounts**: Rotations occur on each cluster's active node independently.

## Exponential backoff

When a rotation fails, the rotation manager re-queues the rotation entry using an
exponential backoff schedule to prevent continuous high-frequency retries from
stressing the system. The backoff schedule also includes a randomized factor
(jitter) to help ensure a more even load distribution. The backoff schedule
has 3 elements:

1. **Initial delay** - the first retry attempt uses a minimum delay of 10 seconds.
1. **Doubling interval** - Subsequent delays double the previous backoff interval
(e.g., 10s, 20s, 40s, 80s) and apply jitter to further stagger the retry attempt.
1. **Maximum cap** - Vault caps the delay for any single retry at 5 minutes (300
seconds).

## Check rotation status

Read the role or credential in the plugin to check the rotation status. The response includes
the fields `last_vault_rotation` which states the timestamp of the last successful rotation
performed by Vault. The field is updated when an automated rotation succeeds and if a manual
rotation occurs between scheduled rotations. The response also includes `next_vault_rotation`
which states the timestamp of the next scheduled automatic rotation. For example, reading
an LDAP static role:

```shell-session
$ vault read ldap/static-cred/my-role
```

```
Key					           Value
---					           -----
last_vault_rotation    2026-04-02T23:30:00Z
next_vault_rotation    2026-04-03T11:30:00Z
ttl                    84600
...
```

## Troubleshooting

Rotation manager logs rotation activity to the Vault server logs such as successful rotations,
rotation failures, retry attempts, and orphaning. Metadata is included in the log statements
to identify the credential being rotated and error information in the case of a failure.

Example log statements for a database secrets engine config may look like:

```
[INFO]  rotation-job-manager: successfully rotated job: rotationID=database/config/my-database
[ERROR] rotation-job-manager: rotation failed, attempting to re-queue: rotationID=database/config/my-database error="connection refused"
[ERROR] rotation-job-manager: orphaning item, please re-register after resolving the error: rotationID=database/config/my-database
```

## Migration from legacy rotation system in plugins

<Warning>

When you unseal Vault the first time after making an upgrade that includes a plugin onboarding
to the rotation manager system, Vault must migrate its rotation entries into the new queueing
system. 

We recommend monitoring the migration process through the static-migration endpoint of the given
plugin until it completes. For example, the LDAP secrets engine static role migration uses the
[`/static-migration`](/vault/api-docs/secret/ldap#static-role-migration) endpoint.

</Warning>

The migration process runs in the background and does not prevent ordinary
operation of the plugin. Vault pauses individual static role rotations until
the role migrates. Once the role migrates, the normal rotation schedule resumes.

See the given plugin's API documentation for more details on managing the migration process.
