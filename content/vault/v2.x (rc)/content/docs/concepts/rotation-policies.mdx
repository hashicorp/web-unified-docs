# Rotation policies
@include 'alerts/enterprise-only.mdx'

Rotation Policies are a set of instructions used by the Rotation Manager to define standardized
error-handling behaviors and lifecycle tracking for onboarded credentials. Rotation Policies
are managed via the Vault API.

Rotation Policies are currently enabled on the following workflows onboarded onto the Rotation Manager:
- LDAP Secrets Enterprise Static Roles
- OS Secrets Static Roles

Plugin administrators can assign a specific rotation policy to a credential when configuring it.
The Rotation Manager then uses these rotation policies to apply tailored retry logic for credentials
and determine how to process or orphan failing rotation entries.

**Note:** Rotation policies are unrelated to [Policies](/vault/docs/concepts/policies) other than sharing
similar names.

Rotation policies are available in Vault version 2.0+. [API docs can be found here](/vault/api-docs/system/policies-rotation).

!> Rotation policies determine when a credential should be transitioned to an "Orphaned" state when it exceeds the maximum retry limits.
Orphaning will halt automated rotation attempts and require manual intervention. Setting these maximum limits should be performed with caution.

## Rotation policy syntax

Rotation Policies are defined in JSON. They allow configuring a set of parameters
that allow the Rotation Manager to determine if a failed rotation should be re-queued with an exponential backoff,
re-queued at its next scheduled rotation, or orphaned entirely.

Rotation Policies support setting the following parameters:

### `max_retries_per_cycle` parameter

- `max_retries_per_cycle` (`int:<required>`) - Defines an explicit max limit for the number of times the
  backoff should be applied during recurring rotation failures for rotation entries using this policy.

### `max_retry_cycles` parameter

- `max_retry_cycles` (`int:<required>`) - Defines an explicit max limit for the number of times the
  Rotation Manager re-queues rotation entries using this policy with their original schedule
  once `max_retries_per_cycle` limit is reached.

#### Example

```json
{
  "max_retry_cycles": 3,
  "max_retries_per_cycle": 3
}
```

This policy will limit the handling of a failed rotation by the Rotation Manager to a maximum of
three exponential backoff attempts within a single rotation cycle (`max_retries_per_cycle`: 3). If
the rotation task fails and reaches this limit, the Rotation Manager will then attempt to re-queue
the rotation entry with its original schedule. The policy allows this re-queue process to occur a total
of three times, which is the explicit limit set (`max_retry_cycles`: 3). If the rotation entry continues
to fail even after all these attempts, the Rotation Manager will determine that the maximum retry limits
have been exceeded. At this point, the Rotation Manager transitions the credential's rotation to an "Orphaned"
state, halting any further automated rotation attempts.

## Default Rotation policy

Vault ships with a default rotation policy that is automatically used if no rotation policy is specified.
The default rotation policy includes the following parameters:

```json
{
  "max_retry_cycles": 3,
  "max_retries_per_cycle": 6
}
```

## Usage

Rotation policies can be assigned when onboarding a credential to the Rotation Manager.
Below is an example of specifying a rotation policy on an LDAP Secrets Enterprise static role:

```
vault write ldap/static-role/alice \
    dn="cn=alice,ou=users,dc=example,dc=com" \
    username="alice" \
    rotation_period="10s" \
    rotation_policy="custom-ldap-policy"
```

If no policy is specified upon initial configuration, the default rotation policy is used.
The rotation policy can also be updated. In the example below, the initial rotation policy is cleared out.
This informs the Rotation Manager to use the default rotation policy for this credential going forward:

```
vault write ldap/static-role/alice \
    dn="cn=alice,ou=users,dc=example,dc=com" \
    username="alice" \
    rotation_period="10s" \
    rotation_policy=""
```

## Exponential Backoff

When a rotation task fails, the Rotation Manager re-queues it using an exponential backoff schedule
to prevent continuous high-frequency retries from stressing the system.

The backoff schedule operates as follows:
- Initial Delay: The first retry attempt uses a minimum delay of 10 seconds.
- Doubling Interval: Subsequent delays double the previous backoff interval (e.g., 10s, 20s, 40s, 80s, etc.).
- Maximum Cap: The delay for any single retry is capped at a maximum of 5 minutes (300 seconds).
- Jitter: A randomized factor (jitter) is applied to the calculated delay to ensure more even load distribution
  across the cluster.

The `max_retries_per_cycle` parameter defines the explicit limit for how many times this exponential backoff
schedule is applied during a single rotation cycle.

## Orphaning

When a rotation entry exceeds the maximum retry limits defined by its policy, the Rotation Manager
transitions the credential to an **Orphaned** state, halting all further automated rotation attempts.
This mechanism is used to prevent the Rotation Manager from being clogged by persistent errors.

Restoring an orphaned rotation entry requires manual intervention from the plugin administrator.

To re-enable automated rotations on a credential:
- resolve the underlying error that caused the credential to be orphaned.
- manually re-register the entry with the Rotation Manager by re-running its original configuration command.
  This action re-registers the entry without changing any configuration data.

For example, to re-register the aforementioned credential for role `alice` with its existing configuration:

```
vault write ldap/static-role/alice \
    dn="cn=alice,ou=users,dc=example,dc=com" \
    username="alice" \
    rotation_period="10s" \
    rotation_policy="custom-ldap-policy"
```

Observability into orphans is available via the Vault API. For more details, please refer to the [API docs](/vault/api-docs/system/rotation-orphans).
