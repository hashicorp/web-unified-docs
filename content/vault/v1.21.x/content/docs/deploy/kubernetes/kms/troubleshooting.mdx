# Troubleshoot the Vault Kubernetes KMS plugin

## Plugin cannot access Vault

If the plugin cannot connect to Vault, check the plugin logs for error messages.
If you deployed the plugin as a static Pod, you can view the logs with the
Kubernetes CLI by providing the name of your static Pod and namespace under
which you deployed it:


```shell-session
$ kubectl logs -n <deploy_namespace> <pod_name>
```

### Incorrect policy configuration

If you see permission denied errors in the plugin logs, verify that the Vault policy
attached to the AppRole includes the required permissions:

```hcl
path "<mount_path>/keys/<key_name>" {
    capabilities = ["read"]
}

path "<mount_path>/encrypt/<key_name>" {
    capabilities = ["update"]
}

path "<mount_path>/decrypt/<key_name>" {
    capabilities = ["update"]
}

path "sys/license/status" {
    capabilities = ["read"]
}
```

This assumes your Transit secrets engine is mounted to the path `transit`. 
If you have chosen a different path for your Transit secrets engine, you will need to put that in the policy instead.

The `sys/license/status` path is required for Vault Enterprise validation. If you
omit this permission, the plugin cannot verify that you are running Vault Enterprise and will not start.

### Incorrect auth mount or AppRole name

If you see authentication errors in the plugin logs, verify the following:

- The `--approle-role-id` flag matches the Role ID for your AppRole in Vault.
- The `--approle-secret-id-path` flag references a valid file system path to a valid Secret ID for the AppRole.
- The static Pod running the plugin has access to the file system path referenced by `--approle-secret-id-path`.
- You enabled the AppRole auth method in Vault at the same mount path that you provided to the plugin.

### Vault is unreachable

If you see connection errors in the plugin logs, verify the following:

- The `--vault-address` flag points to a reachable Vault server.
- Network policies or firewalls allow traffic from the control plane node to Vault.
- If you use TLS, make sure one of the following is true:
  - The `--vault-ca` flag points to a valid CA certificate.
  - The system trust store includes a valid CA for the Vault server.

### Expired AppRole Secret ID

If the plugin worked previously but authentication starts failing, the AppRole
Secret ID may have expired.

1. Generate a new Secret ID in Vault and replace the contents of the file at
   `--approle-secret-id-path`:

   ```shell-session
   $ vault write -f auth/approle/role/<role_name>/secret-id

1. Copy the new Secret ID to the file on each control plane node.

1. Restart the plugin after updating the Secret ID so it reads the updated file
   that uses the new Secret ID.

### Vault Enterprise validation failing

The Vault Kubernetes KMS plugin requires Vault Enterprise. The following error
message indicates that the plugin connected to a Vault Community instance instead:

<CodeBlockConfig hideClipboard>

```shell
vault Community Edition detected - vault-kube-kms requires Vault Enterprise

## API server cannot access the plugin

If the Kubernetes API server cannot communicate with the plugin, Kubernetes reports
errors when encrypting or decrypting data. Check the `kube-apiserver` logs for KMS-related
error messages.

The Kubernetes API server communicates with the KMS plugin over a Unix domain
socket with 0770 permissions (owner and group). To communicate successfully, the
API server process must run as `root` or with the same Unix group assignment as
the plugin.

### Socket address mismatch

The `endpoint` field in the Kubernetes `EncryptionConfiguration` must match the
`--listen-address` flag passed to the plugin.

Verify that both values point to the same Unix socket path. For example, if your
`EncryptionConfiguration` contains the following:

```yaml
providers:
  - kms:
      apiVersion: v2
      name: vault-kms
      endpoint: unix:///var/run/kmsplugin/kms.sock
```

You must start your KMS plugin with the same address:

```shell-session
--listen-address=unix:///var/run/kmsplugin/kms.sock
```

If the API endpoint and listen addresses match and you still see socket errors,
verify that both the `kube-apiserver` Pod and the plugin Pod have mounted the directory containing the socket.
If either Pod fails to mount the directory, that Pod won't be able to access the Unix socket for
the plugin.

### Missing or invalid configuration flags

Read the plugin's logs in the static Pod with:
```shell-session
$ kubectl logs -n <NAMESPACE> <POD_NAME>
```

Look for validation errors at the start of the logs to see if a configuration flag or environment variable was 
passed incorrectly and causing the plugin to fail to start.

## Encryption or decryption errors

If encryption or decryption operations fail after the plugin connects to Vault,
the Transit key configuration may be incompatible.

### Key version constraints

If decryption fails for data encrypted with an older key version, verify the
`min_decryption_version` setting on the Transit key in Vault:

```shell-session
$ vault read transit/keys/<KEY_NAME>
```

If `min_decryption_version` is higher than the key version used to encrypt the data,
Vault rejects decryption requests. Lower the `min_decryption_version` or re-encrypt
the data with a newer key version.

~> **Warning:** Changing `min_decryption_version` can prevent Kubernetes from reading
data encrypted with older key versions. Verify that all encrypted data uses a key
version at or above the new minimum before increasing this value.

Similarly, if `min_encryption_version` is set, verify that the plugin is using a
key version at or above the minimum.

## Re-encrypting existing data

After rotating the Transit key in Vault, existing encrypted data in etcd remains
encrypted with the old key version. Kubernetes does not automatically re-encrypt
existing data.

To re-encrypt existing Secrets with the new key version, perform a no-op write
to force Kubernetes to re-encrypt them:

```shell-session
$ kubectl get secrets --all-namespaces -o json | kubectl replace -f -
```

For other resource types configured in your `EncryptionConfiguration`, repeat this
process for each resource type.

For large clusters or complex migrations, refer to the Kubernetes documentation on
[encrypting confidential data at rest](https://kubernetes.io/docs/tasks/administer-cluster/encrypt-data/)
for guidance on storage migration strategies.
