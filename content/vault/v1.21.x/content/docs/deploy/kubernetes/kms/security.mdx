# Security model for Vault Kubernetes KMS

## Overview

This page explains the security model of Vault Kubernetes KMS and the security controls available to end users. 
It describes the architecture, trust boundaries, threat model, and provides best practices for deploying and operating the 
process securely in production Kubernetes environments.

## Key concepts

Vault Kubernetes KMS enables envelope encryption of Kubernetes secrets (and other etcd-stored resources) using HashiCorp Vault's 
Transit secrets engine as the external Key Encryption Key (KEK) provider.

## Envelope encryption

Envelope encryption is a two-layer encryption scheme that encrypts data with a
Data Encryption Key (DEK) then encrypts the DEK itself with a second Key
Encryption Key (KEK). Envelope encryption efficiently encrypts large amounts of
data while enabling centralized key management and rotation.

### Kubernetes KMS v2 protocol

The Kubernetes KMS v2 protocol defines the interface between a Kubernetes API
server and an external key management service like Vault. Unlike KMS v1, which
generated a new DEK for every encryption operation, KMS v2 uses a DEK seed
approach.

- The API server generates a single DEK seed at startup
- Vault Kubernetes KMS encrypts this seed using the external KEK (from Vault Transit)
- The API server derives single-use DEKs from the seed using HKDF (HMAC-based Key Derivation Function)
- Vault Kubernetes KMS is only called once at startup and when KEK rotation is detected

### DEK seeds and KEK management

The Data Encryption Key seed is a cryptographic secret generated by the Kubernetes API server. 
The API server combines this seed with random data through HKDF to derive unique DEKs for each encryption operation. 
The seed is encrypted by Vault Kubernetes KMS and stored alongside encrypted data in etcd. Vault Kubernetes KMS does not store this data.

### Key Encryption Key (KEK)

Vault manages the KEK with the Transit plugin, which encrypts and decrypts the
DEK seeds. The KEK never leaves Vault and all cryptographic operations involving
the KEK happen within the Transit plugin.

### The Transit plugin

The Transit plugin is a secrets engine that provides encryption as a service
(EaaS) cryptographic operations without exposing the underlying keys. The engine
supports key rotation, convergent encryption, and multiple key types including
AES-256-GCM96.

### Unix Domain Socket

The communication channel between the Kubernetes API server and Vault Kubernetes KMS. 
Unix sockets provide process-to-process communication with kernel-enforced access controls based on filesystem permissions.

## Architecture

The Kubernetes API server and the KMS plugin use Unix sockets to provide
process-to-process communication with kernel-enforced access controls based on
filesystem permissions.

### Component Description

#### Vault Kubernetes KMS (gRPC Server)

The core component that implements the Kubernetes [KMS v2](https://kubernetes.io/docs/tasks/administer-cluster/kms-provider/#implementing-a-kms-plugin) interface. It:

- **Listens** on a Unix domain socket for gRPC requests from the API server
- **Authenticates** to Vault using AppRole authentication
- **Forwards** encrypt/decrypt operations to Vault Transit
- **Caches** the Vault client with automatic token renewal
- **Reports** health status and current KEK version to Kubernetes

#### Unix Domain Socket

Vault Kubernetes KMS creates a Unix socket at a configurable path (default: `/tmp/vault-kube-kms.socket`) with 
permissions `0770` (owner and group read/write/execute). This restricts access to:
- The `vault-kube-kms` process (owner)
- Processes in the same group

#### Vault Client

A cached client that:
- Authenticates using AppRole (role ID + secret ID from filesystem)
- Automatically renews tokens before expiration
- Re-authenticates if tokens exceed max TTL

#### AppRole Authentication

Vault Kubernetes KMS uses Vault's AppRole auth method:
- **Role ID**: Non-sensitive identifier, can be configured via flag or environment variable
- **Secret ID**: Sensitive credential read from a filesystem path, rather than passed directly

### Security Controls

| Control | Implementation |
|---------|----------------|
| **Socket Permissions** | Unix socket created with umask `0007` restricting access to owner+group |
| **TLS to Vault** | TLS enabled by default; custom CA certificate supported; SNI configurable |
| **Authentication** | AppRole with secret ID from protected filesystem path |
| **Token Management** | Automatic renewal; re-authentication on token expiry |
| **Non-root Container** | Runs as UID 65532 (nonroot) in container image; UBI image for OpenShift |
| **Input Validation** | Strict validation of key IDs, configuration values, and Vault responses |
| **Enterprise Validation** | Verifies Vault Enterprise license (not Community Edition) |
| **Logging** | Structured JSON logging with configurable levels; sensitive data redacted |
| **Metrics** | Prometheus endpoint with gRPC latency histograms |

## Threat Model

The design for Vault Kubernetes KMS addresses the following categories of threats:

| Threat | Description | Key mitigations |
|--------|-------------|------------------|
| **Unauthorized socket access** | A process on the control plane node attempting to communicate with Vault Kubernetes KMS without authorization. | Restricted socket permissions, non-root execution, and limited node access. |
| **Network interception** | An attacker intercepting traffic between Vault Kubernetes KMS and Vault. | TLS enabled by default with support for custom CA certificates and SNI configuration. |
| **Key exposure** | Exposing the Transit encryption key (KEK) | The KEK never leaves Vault. Vault Kubernetes KMS only handles ciphertext and plaintext, never raw key material. |
| **Data replay or rollback** | An attacker substitutes older encrypted data. | Key version tracking allows Kubernetes to detect stale encryption and trigger re-encryption. |
| **Vault unavailability** | Vault becoming unavailable | Kubernetes KMS v2 means the API server caches the DEK seed locally, reducing dependency on Vault during normal operation. |
| **Information leakage** | Sensitive data appearing in logs or metrics. | Structured logging with sensitive value redaction. Metrics expose only latency and counters. |
| **Resource exhaustion** | Excessive requests overwhelm Vault Kubernetes KMS. | Client-side caching of connection-handling, and configurable request timeouts. |

---

## Threat Model Exclusions

Vault Kubernetes KMS does not protect against threats that require infrastructure-level controls.
Kubernetes cluster administrators must ensure that access to their control plane nodes are properly secured, 
and Vault cluster administrators must ensure that Vault cluster access is protected with appropriate policy.

## Recommendations For Secure Use

### Deployment

1. **Deploy as a Static Pod**: Run Vault Kubernetes KMS as a static Pod on each control plane node.

2. **Use Minimal Container Image**: Deploy using the distroless or UBI-based images that contain only the process binary and necessary certificates.

3. **Restrict Node Access**: Limit SSH and console access to control plane nodes to authorized administrators only.

4. **Network Policies**: Apply Kubernetes network policies to restrict the `vault-kube-kms` process's egress to only the Vault endpoints.

### Vault Configuration

1. **Use Minimum Required Policy**: Grant only the permissions required:
   ```hcl
   path "transit/encrypt/kms-key" {
     capabilities = ["create", "update"]
   }
   path "transit/decrypt/kms-key" {
     capabilities = ["create", "update"]
   }
   path "transit/keys/kms-key" {
     capabilities = ["read"]
   }
   path "sys/license/status" {
     capabilities = ["read"]
   }
   ```

2. **Use AES-256-GCM96**: Configure the Transit key with the `aes256-gcm96` type for authenticated encryption.

3. **Enable Key Versioning**: Do not disable key versioning; it is required for proper key rotation support.

4. **Rotate Keys Regularly**: Rotate the Transit key at least every 90 days per Kubernetes project recommendations.

5. **Use Vault Namespaces**: In multi-tenant environments, isolate the KMS Transit key in a dedicated Vault namespace.

### Authentication

1. **Protect Secret ID**: Mount the AppRole Secret ID as a file with restricted permissions (mode 0400).

2. **Use Short Token TTLs**: Configure AppRole with short token TTLs.

3. **Rotate Secret IDs**: Periodically rotate AppRole secret IDs and update the mounted secret.

### TLS Configuration

1. **Always Use TLS**: Never disable TLS verification (`--tls-skip-verify`) in production.

2. **Provide CA Certificate**: If not using the CA in the system trust store, provide the CA certificate for Vault via `--tls-ca-file`.

3. **Configure SNI**: If Vault is behind a load balancer with a different hostname, configure `--tls-sni`.

### Monitoring and Auditing

1. **Enable Vault Audit Logging**: Enable audit logging in Vault to record all Transit operations.

2. **Monitor Metrics**: Scrape the Prometheus `/metrics` endpoint for:
   - `rpc.server.call.duration` histograms (ensure latency stays within KMS v2 requirements)
   - Error rates on Status/Encrypt/Decrypt operations

3. **Alert on Failures**: Configure alerts for:
   - `vault-kube-kms` process health check failures
   - Authentication failures to Vault

4. **Review Logs**: Regularly review `vault-kube-kms` process logs for authentication failures or unexpected errors.

### Disaster Recovery

1. **Backup Transit Key**: Ensure the Transit key can be recovered; consider Vault's key export functionality for disaster recovery (with appropriate safeguards).

2. **Document Recovery Procedures**: Document how to recover if Vault Kubernetes KMS or Vault becomes unavailable.

3. **Test Recovery**: Periodically test recovery procedures including API server restart with Vault Kubernetes KMS.

### Upgrade Path

1. **Test Upgrades in Non-Production**: Test Vault Kubernetes KMS upgrades in a non-production cluster first.

2. **Monitor During Upgrades**: Watch for authentication failures or latency increases during rolling upgrades.

3. **Maintain Backward Compatibility**: Ensure Vault can decrypt data encrypted with older key versions by configuring the `min_decryption_version` field.
