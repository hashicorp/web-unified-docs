# Key Rotation

The Vault Kubernetes KMS plugin relies on the Vault Transit secrets engine
for cryptographic operations. Key rotation is performed **in Vault**, and does
not require changes to the Kubernetes API server configuration.

~> **Warning:** Changing the Transit key’s `min_decryption_version` can prevent Kubernetes
from reading data encrypted with older key versions.

## Rotate the Transit key

The Vault Kubernetes KMS plugin uses a Vault Transit key to encrypt data
encryption keys (DEKs) for Kubernetes envelope encryption.

To rotate the key, call the Transit rotate endpoint for the key configured in the
plugin.

```bash
vault write -f transit/keys/kms/rotate
```

Vault creates a new version of the key while retaining previous versions for
decryption.

After the rotation occurs, the plugin automatically detects the new Transit key
version. Within a few minutes of rotation, the plugin updates the
active key version used for encryption. See "Understanding key_id and Key Rotation" 
section of the Kubernetes [docs](https://kubernetes.io/docs/tasks/administer-cluster/kms-provider/#developing-a-kms-plugin-gRPC-server-notes-kms-v2) for more details.

A new DEK is encrypted using the latest key version, while existing encrypted
DEKs remain decryptable using older key versions. No restart of the plugin or
the Kubernetes API server is required.

During this process, the plugin emits log entries indicating that a new Transit
key version was observed and the encryption operation succeeded using the new key version.
