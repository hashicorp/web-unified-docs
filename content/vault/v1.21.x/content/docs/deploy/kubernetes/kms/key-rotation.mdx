# Key Rotation

Vault Kubernetes KMS relies on the Vault Transit secrets engine
for cryptographic operations. A Vault cluster administrator rotates the key **in Vault**, so there 
is no need to update the Kubernetes API server configuration to encrypt with the most recent key version.

~> **Warning:** Changing the Transit key’s `min_decryption_version` can prevent Kubernetes
from reading data encrypted with older key versions. Ensure your Transit key's `min_decryption_version` 
includes the oldest key version used by your Kubernetes cluster.

## Rotate the Transit key

Vault Kubernetes KMS uses a Vault Transit key to encrypt data
encryption keys (DEKs) for Kubernetes envelope encryption.

To rotate the key, call the Transit rotate endpoint with the key name passed to the
`vault-kube-kms` process.

```bash
vault write -f transit/keys/kms/rotate
```

Vault creates a new version of the key while retaining previous versions for
decryption.

After you rotate the key in Vault, Vault Kubernetes KMS automatically detects the new Transit key
version. Within a few minutes of rotation, Vault Kubernetes KMS updates the
active key version used for encryption. See "Understanding key_id and Key Rotation" 
section of the Kubernetes [docs](https://kubernetes.io/docs/tasks/administer-cluster/kms-provider/#developing-a-kms-plugin-gRPC-server-notes-kms-v2) for more details on the timing.

The Kubernetes API server encrypts the new DEK using the latest key version, while existing encrypted
DEKs remain decryptable using older key versions. You do not need to restart Vault Kubernetes KMS or
the Kubernetes API server.

During this process, Vault Kubernetes KMS emits log entries indicating that it observed a new Transit
key version and the encryption operation succeeded using the new key version.
