# Configuration

This page documents the configuration parameters for Vault Kubernetes KMS. 
These parameters control how Vault Kubernetes KMS exposes its KMS v2 gRPC
endpoint to Kubernetes, connects to Vault, authenticates, selects a Transit key,
and emits logs and metrics.

You provide configuration to Vault Kubernetes KMS at startup using command-line flags.
It validates configuration on startup and exits if required parameters
are missing or invalid.

---

## Parameters

- `listen-address` `(string: "unix:///tmp/vault-kube-kms.socket")` – The
  Unix domain socket address where Vault Kubernetes KMS listens for requests from
  the Kubernetes API server. The `kube-apiserver` process must have filesystem access
  to this socket in order to communicate with the `vault-kube-kms` process.

  Environment Variable : `VAULT_KUBE_KMS_LISTEN_ADDRESS`

- `vault-address` `(string: <required>)` – The HTTP(S) API address of the Vault
  server. Vault Kubernetes KMS connects to this address to authenticate and perform cryptographic
  operations using the Vault Transit secrets engine.

  This value must be a non-empty string and the node where the `vault-kube-kms` process runs must be able to reach this address.

  Environment Variable : `VAULT_KUBE_KMS_VAULT_ADDRESS`

- `vault-namespace` `(string: "")` – The Vault namespace where the Transit secrets engine is deployed.
  If unset, Vault Kubernetes KMS will assume Transit is in the root namespace.

  You must set this parameter when using Vault Enterprise with namespaces enabled.

  Environment Variable: `VAULT_KUBE_KMS_VAULT_NAMESPACE`

- `transit-mount` `(string: "transit")` – The mount path to the Vault Transit secrets engine. 
  Vault Kubernetes KMS uses this to craft the request to Vault to encrypt and decrypt data encryption keys (DEKs).

  This value must be a non-empty string and must reference an existing Transit
  mount in Vault.

  Environment Variable: `VAULT_KUBE_KMS_TRANSIT_MOUNT`

- `transit-key` `(string: "")` – The name of the Vault Transit key that Vault Kubernetes KMS uses to 
  craft the request to Vault to encrypt and decrypt DEKs. 
  
  The key must already exist in Vault.

  Vault Kubernetes KMS’s Vault policy must grant permission to use this key for encryption
  and decryption operations.

  Environment Variable: `VAULT_KUBE_KMS_TRANSIT_KEY`

- `zap-log-level` `(string: "info")` – Log verbosity level.
  Supported values (in order of descending detail) are `trace`, `debug-extended`,
  `debug`, `info`, and `error`

  Environment Variable: `VAULT_KUBE_KMS_LOG_LEVEL`

- `auth-method` `(string: "appRole")` – The Vault authentication method used by Vault Kubernetes KMS.
  Supported values are `appRole`.

  Environment Variable: `VAULT_KUBE_KMS_AUTH_METHOD`

- `tls-ca-file` `(string: "")` – Path to a CA certificate file for validating Vault's TLS certificate.
  If not set, Vault Kubernetes KMS will use the system trust store.

  Environment Variable: `VAULT_KUBE_KMS_TLS_CA_FILE`

- `tls-skip-verify` `(bool: false)` – Skip TLS certificate verification when connecting to Vault.
  Not for production use.

  Environment Variable: `VAULT_KUBE_KMS_TLS_SKIP_VERIFY`

- `tls-sni` `(string: "")` – SNI (Server Name Indication) to use when connecting to
  Vault via TLS

  Environment Variable: `VAULT_KUBE_KMS_TLS_SNI`

## Authentication parameters

### AppRole

When you configure Vault Kubernetes KMS to authenticate with AppRole, the following parameters are required.

- `approle-role-id` `(string: "")` – Vault Kubernetes KMS uses the AppRole Role ID to identify itself when authenticating
  to Vault. This value is not sensitive so you can provide it directly.

  Environment variable: `VAULT_KUBE_KMS_APPROLE_ROLE_ID`

- `approle-secret-id-path` `(string: "")` – The filesystem path to a file
  containing the AppRole Secret ID. This value is sensitive and you should protect it
  using appropriate filesystem permissions.

  Vault Kubernetes KMS reads the Secret ID from this file at startup.

  Environment variable: `VAULT_KUBE_KMS_APPROLE_SECRET_ID_PATH`

## Metrics and telemetry parameters

- `metrics-port` `(int: 9090)` – The local TCP port where Vault Kubernetes KMS exposes a
  Prometheus-compatible `/metrics` endpoint. The endpoint binds to `localhost`.

  If unset, Vault Kubernetes KMS does not expose a metrics endpoint.

  Environment variable: `VAULT_KUBE_KMS_METRICS_PORT`

- `enable-runtime-metrics` `(bool: false)` – Controls whether Go runtime and
  process-level metrics are exposed alongside Vault Kubernetes KMS metrics.

  When enabled, the `/metrics` endpoint includes metrics such as garbage
  collection statistics, goroutine counts, and memory usage.

  Environment variable: `VAULT_KUBE_KMS_ENABLE_RUNTIME_METRICS`

## Minimal configuration example

```text
--listen-address=/var/run/kms/vault-kms.sock
--vault-address=https://vault.example.com:8200
--transit-mount=transit
--transit-key=kubernetes-etcd
--auth-method=approle
--approle-role-id=example-role-id
--approle-secret-id-path=/var/run/secrets/vault/secret-id
```
