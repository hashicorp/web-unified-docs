#### Issue

Azure is eventually consistent, so internal state propagation may lag after an object is created or modified. When Vault returns a credential for a dynamic role, it must create an application, instantiate a service principal and then provision a new credential. This operation may intermittently fail if the service principal is in an inconsistent state with an error message resembling:

```
The service principal cannot be created, updated, or restored because the service principal name {guid} is already in use.
```

#### Workaround

The Azure secrets engine now makes use of more efficient API calls to better avoid inconsistent states. It is recommended to upgrade to the latest patch release of Vault in order to resolve the issue.
