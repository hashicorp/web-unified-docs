#### Issue

Azure is eventually consistent, so internal state propagation may lag when Vault
creates or modifies an object. When Vault returns a credential for a dynamic
role, it must create an application, instantiate a service principal, and then
provision a new credential.

When the service principal is in an inconsistent state, Vault may intermittently
return an error similar to the following when the credential requests fails:

```text
The service principal cannot be created, updated, or restored because the service principal name {guid} is already in use.
```

#### Workaround

Ugrade to latest patch release of Vault. As of the latest Vault versions, the
Azure secrets engine uses of more efficient API calls to better avoid
inconsistent states.
