Microsoft SQL Server availability groups do not replicate server-level logins.

When using the Vault database secrets engine with a Microsoft SQL Server
Availability Group (AG), executing a [`rotate-root`](/vault/api-docs/secret/databases#rotate-root-credentials)
operation will update the login/user password on the instance configured in
the database secrets engine. This leaves the other instances in the availability
group out of sync and can cause authentication failures if a failover occurs.

If you use Microsoft SQL Server Availability Groups (AG) there are other
workflows to consider.

1. Configure Microsoft SQL Server with Active Directory authentication, and use
   the [LDAP secrets engine](/vault/docs/secrets/ldap) to manage the Active
   Directory user credentials. You can use the LDAP credentials to log in
   across all instances in the availability group.

1. Rotate the login/user password for the database secrets engine configuration
   outside of Vault on all instances in the availability group, for example by
   using Terraform or Ansible. After updating the passwords, update the Vault
   configuration.
