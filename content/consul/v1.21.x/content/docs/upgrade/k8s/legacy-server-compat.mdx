# Migrate to Consul dataplane with legacy server compatibility

This topic describes how to upgrade Consul dataplane to v1.8.x when your Consul servers do not yet advertise full dataplane feature support. The `-legacy-server-compat` flag lets you upgrade the dataplane layer first so that you can progress to a Consul long-term support (LTS) release without disrupting mesh traffic.

## Background

When you run Consul 1.18.x with consul-k8s 1.4.x and Consul dataplane 1.4.x, both standard upgrade paths can block your progress:

- When you upgrade the dataplane first, Consul dataplane 1.8.x rejects the connection because older servers do not advertise the `DATAPLANE_FEATURES_ENVOY_BOOTSTRAP_CONFIGURATION` feature.
- When you upgrade Consul first, the older dataplane lacks the Envoy version that the newer Consul servers require, so mesh traffic stops until you upgrade the dataplane.

The `-legacy-server-compat` flag provides a temporary compatibility mode that bypasses the dataplane feature check so that you can connect Consul dataplane 1.8.x to an older Consul server during a migration.

## Requirements

- Consul dataplane v1.8.x.
- consul-k8s v1.4.x, which is the only supported pairing with Consul 1.18.x. Do not upgrade consul-k8s until you upgrade your Consul servers.
- An existing Consul server that you intend to upgrade to a supported LTS release.

## Migration paths

Choose a migration path based on whether you need to maintain mesh traffic during the upgrade.

| Path | Mesh traffic | DNS | Pod restarts | Flag required |
| --- | --- | --- | --- | --- |
| Dataplane-first | Available throughout | Available throughout | 2 | Yes (`-legacy-server-compat`) |
| Consul-first | Stops during the Consul upgrade | Available throughout | 1 | No |

During either migration window, central telemetry and merged Prometheus metrics are unavailable.

### Option 1: Dataplane-first migration

We recommend the dataplane-first path when you require zero mesh traffic disruption.

1. Upgrade Consul dataplane from v1.4.x to v1.8.x with the `-legacy-server-compat=true` flag. This step requires a pod restart.
1. Perform a rolling upgrade of Consul from v1.18.x to a supported LTS release.
1. Upgrade consul-k8s from v1.4.x to v1.8.x.
1. Remove the `-legacy-server-compat=true` flag. This step requires a second pod restart.

Mesh traffic remains available throughout this migration. Central telemetry remains disabled until you remove the flag.

### Option 2: Consul-first migration

The Consul-first path minimizes dataplane restarts, but mesh traffic stops during the Consul rolling upgrade window.

1. Perform a rolling upgrade of Consul from v1.18.x to a supported LTS release. The dataplane remains at v1.4.x.
1. Perform a rolling upgrade of Consul dataplane from v1.4.x to v1.8.x.
1. Upgrade consul-k8s from v1.4.x to v1.8.x.

DNS resolution remains available throughout this migration because the dataplane DNS proxy uses an independent gRPC stream. Mesh traffic stops during the Consul rolling upgrade because the newer servers reject the Envoy version that the older dataplane bundles. Mesh traffic resumes after you upgrade the dataplane to v1.8.x.

## Enable legacy server compatibility

To enable the compatibility mode, set the `-legacy-server-compat` flag when you start Consul dataplane:

```shell-session
$ consul-dataplane -legacy-server-compat=true
```

Alternatively, set the `DP_LEGACY_SERVER_COMPAT` environment variable:

```shell-session
$ export DP_LEGACY_SERVER_COMPAT=true
```

When you enable the flag, Consul dataplane logs the features that the compatibility mode disables. If the connected server already supports the full dataplane feature set, Consul dataplane lifts the compatibility guards and recommends that you remove the flag.

## Next steps

After you upgrade your Consul servers and confirm that the dataplane connects without warnings, remove the `-legacy-server-compat` flag and restart the affected pods.
