# Multiport services

In a Consul service mesh, services can be configured to use multiple ports. This allows you to run different types of traffic through the same service while still benefiting from Consul's service discovery and traffic management features.

## Overview

Many applications require multiple ports to function properly. For example, a web application might have one port for HTTP traffic and another for gRPC traffic. With Consul Connect, you can configure your services to use multiple ports while still leveraging the benefits of service mesh features such as secure communication, traffic routing, and observability.

## Configuring multiport services

To use multiport services in Consul Connect, you need to ensure that your service definitions include the necessary port information. This typically involves defining multiple ports in your service configuration and ensuring that your proxy configuration is set up to handle traffic on those ports.

The specific requirements for multiport services varies depending on your deployment environment.There are two specific cases to consider when configuring multiport services: non-transparent proxy and transparent proxy. Each case has its own configuration requirements and considerations, which we will discuss in the following sections.

### Non-transparent proxy

In Non-Transparent proxy mode, downstream services explicitly configure upstreams by defining a `destination_name`, `local_bind_port`, and specifically to the multiport use case - a `destination_port`. This allows the downstream service to specify which port on the upstream service it wants to connect to, and the proxy will handle routing the traffic accordingly. Check out these details further in the [upstream configuration reference](docs/reference/proxy/connect-proxy#upstream-configuration-reference).

An example configuration for a non-transparent proxy service with multiport upstream services is shown below. In this example, the `web` service is configured to connect to two different ports on the `api` service (9090 and 8080) and one port on the `db` service (9191).

<CodeBlockConfig language="json" highlight="12-21">

```json
{
  "name": "web",
  "port": 8080,
  "connect": {
    "sidecar_service": {
      "proxy": {
        "upstreams": [
          {
            "destination_name": "db",
            "local_bind_port": 9191
          },
          {
            "destination_name": "api",
            "destination_port": 9090,
            "local_bind_port": 9192
          },
          {
            "destination_name": "api",
            "destination_port": 8080,
            "local_bind_port": 9193
          }
        ],
        "config": {
          "handshake_timeout_ms": 1000
        }
      }
    }
  }
}
```
