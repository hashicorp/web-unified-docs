# Delegate API gateway authorization to an external service on virtual machines

@include 'alerts/enterprise-alert.mdx'

This topic describes how to delegate authorization for requests to Consul API gateways to an external service on virtual machines (VM). If your services are deployed to Kubernetes-orchestrated containers, refer to [Delegate API gateway authorization to an external service on Kubernetes](/consul/docs/north-south/api-gateway/secure-traffic/ext-authz/k8s).

## Overview

The [external authorization Envoy extension](/consul/docs/reference/proxy/extensions/ext-authz) (`builtin/ext-authz`) directs Envoy to call an external authorization service before it forwards a request to an upstream. The extension's `ProxyType` field determines where Consul attaches it. Set `ProxyType` to `connect-proxy` to enforce authorization at a service's sidecar proxy for east-west traffic. Set `ProxyType` to `api-gateway` to attach the extension to the API gateway itself so that Consul applies external authorization to north-south traffic that the gateway routes to services in the mesh. This topic describes the `api-gateway` case.

Applying the extension to the gateway's service defaults configuration entry is a prerequisite for external authorization. None of the following controls take effect until you apply the extension.

After you apply the extension, you can control external authorization at two levels:

- Gateway-wide: The [`ExtAuthz` block in the API gateway configuration entry](/consul/docs/reference/config-entry/api-gateway#extauthz) sets the default authorization behavior for all routes attached to the gateway.
- Per-route: The [`ExtAuthz` filter in an HTTP route configuration entry](/consul/docs/reference/config-entry/http-route#rules-filters-extauthz) overrides the gateway-wide default for the routes it matches.

After you apply the extension, external authorization is enabled for every route attached to the gateway. Complete the following steps to configure it:

1. [Apply the external authorization extension to the API gateway's service defaults configuration entry.](#apply-the-extension-to-the-api-gateway) This step enables external authorization for all routes attached to the gateway.
1. Optionally, [change the gateway-wide default](#set-the-gateway-wide-authorization-behavior) so that external authorization is opt-in.
1. Optionally, [override external authorization for specific routes](#override-the-authorization-behavior-for-specific-routes) in the HTTP route configuration entries.

## Requirements

- Consul Enterprise 2.0.2 or later
- An API gateway deployed and configured to route to one or more mesh services
- An external authorization service that implements the Envoy external authorization gRPC or HTTP API

## Apply the extension to the API gateway

Add the external authorization extension to the service defaults configuration entry for your API gateway and set `ProxyType` to `api-gateway`.

The `api-gateway` `ProxyType` allows you to configure [`Target.URI`](/consul/docs/reference/proxy/extensions/ext-authz#arguments-config-grpcservice-target-uri) to point at an authorization service on a remote host.

<CodeBlockConfig filename="gateway-service-defaults.hcl">

```hcl
Kind = "service-defaults"
Name = "api-gateway"
Protocol = "http"
EnvoyExtensions = [
  {
    Name = "builtin/ext-authz"
    Arguments = {
      ProxyType = "api-gateway"
      Config = {
        GrpcService = {
          Target = {
            URI = "authz.example.com:9191"
          }
        }
        AllowedHeaders = [
          {
            Prefix = "x-"
          }
        ]
        DisallowedHeaders = [
          {
            Exact      = "authorization"
            IgnoreCase = true
          }
        ]
      }
    }
  }
]
```

</CodeBlockConfig>

For the full list of configurable fields, including request and response header controls, refer to the [external authorization extension configuration reference](/consul/docs/reference/proxy/extensions/ext-authz).

Write the configuration entry to Consul:

```shell-session
$ consul config write gateway-service-defaults.hcl
```

## Set the gateway-wide authorization behavior

External authorization is enabled by default for all routes attached to the gateway. This step is optional and only necessary when you want to change that default.

To enable external authorization only for specific routes, use the [`ExtAuthz` block in the API gateway configuration entry](/consul/docs/reference/config-entry/api-gateway#extauthz) to set `ExtAuthz.Enabled` to `false`, then enable it individually for the desired routes as shown in the next section.

<CodeBlockConfig filename="api-gateway.hcl">

```hcl
Kind = "api-gateway"
Name = "api-gateway"

ExtAuthz = {
  Enabled = false
}

Listeners = [
  {
    Name     = "http-listener"
    Port     = 8443
    Protocol = "http"
  }
]
```

</CodeBlockConfig>

Write the configuration entry to Consul:

```shell-session
$ consul config write api-gateway.hcl
```

## Override the authorization behavior for specific routes

Use the [`ExtAuthz` filter in an HTTP route configuration entry](/consul/docs/reference/config-entry/http-route#rules-filters-extauthz) to force external authorization on or off for the routes that the rule matches. The route-level setting takes precedence over the gateway-wide default.

In the following example, external authorization is enabled for requests to `/admin` even when the gateway-wide default is opt-in:

<CodeBlockConfig filename="http-route.hcl">

```hcl
Kind = "http-route"
Name = "admin-route"

Parents = [
  {
    Kind = "api-gateway"
    Name = "api-gateway"
  }
]

Rules = [
  {
    Matches = [
      {
        Path = {
          Match = "prefix"
          Value = "/admin"
        }
      }
    ]
    Filters = {
      ExtAuthz = {
        Enabled = true
      }
    }
    Services = [
      {
        Name = "admin"
      }
    ]
  }
]
```

</CodeBlockConfig>

Write the configuration entry to Consul:

```shell-session
$ consul config write http-route.hcl
```

After Consul applies the configuration, the API gateway calls the external authorization service according to the authorization behavior you configured before it forwards matching requests to upstream services.
