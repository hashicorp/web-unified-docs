# Delegate API gateway authorization to an external service on Kubernetes

@include 'alerts/enterprise-alert.mdx'

This topic describes how to delegate authorization for requests to Consul API gateways to an external service on Kubernetes-orchestrated networks. If your services are deployed to virtual machines, refer to [Delegate API gateway authorization to an external service on virtual machines](/consul/docs/north-south/api-gateway/secure-traffic/ext-authz/vm).

## Overview

The [external authorization Envoy extension](/consul/docs/reference/proxy/extensions/ext-authz) (`builtin/ext-authz`) instructs Envoy to call an external authorization service before it forwards a request to an upstream. 

Consul uses the extension's `ProxyType` field to determine the external service to use for the authentication:

- When `ProxyType` is set to `connect-proxy`, authorization is enforced at a service's sidecar proxy level. This is used for east-west traffic.
- When `ProxyType` is set to `api-gateway` the extension is applied to the API gateway itself and an external authorization is applied to the requests received by the API gateway. This is used for north-south traffic.

This topic describes the `api-gateway` case.

Applying the extension to the gateway through the gateway's `ServiceDefaults` is a prerequisite for external authorization. None of the following controls take effect until you apply the extension.

After you apply the extension, you can control external authorization at two levels:

- Gateway-wide: The `consul.hashicorp.com/ext-authz` annotation on the `Gateway` resource sets the default authorization behavior for all routes attached to the gateway.
- Per-route: A [`RouteAuthFilter`](/consul/docs/reference/k8s/api-gateway/routeauthfilter) referenced by an `HTTPRoute`, or the `consul.hashicorp.com/ext-authz` annotation on an `HTTPRoute`, overrides the gateway-wide default.

Complete the following steps to configure external authorization:

1. [Apply the external authorization extension to the API gateway's `ServiceDefaults` resource.](#apply-the-extension-to-the-api-gateway) This step enables external authorization for all routes attached to the gateway.
1. Optionally, [change the gateway-wide default](#set-the-gateway-wide-authorization-behavior) so that external authorization is opt-in.
1. Optionally, [override external authorization for specific routes](#override-the-authorization-behavior-for-specific-routes) with a `RouteAuthFilter` or annotation.

## Requirements

- Consul Enterprise 2.0.2 or later
- An API gateway deployed and configured to route to one or more mesh services
- An external authorization service that implements the Envoy external authorization gRPC or HTTP API

## Apply the extension to the API gateway

Add the external authorization extension to the `ServiceDefaults` resource for your API gateway and set `ProxyType` to `api-gateway`.

The `api-gateway` `ProxyType` allows you to configure [`Target.URI`](/consul/docs/reference/proxy/extensions/ext-authz#arguments-config-grpcservice-target-uri) to point at an authorization service on a remote host.

<CodeBlockConfig filename="gateway-service-defaults.yaml">

```yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: ServiceDefaults
metadata:
  name: api-gateway
spec:
  protocol: http
  envoyExtensions:
  - name: builtin/ext-authz
    arguments:
      ProxyType: api-gateway
      Config:
        GrpcService:
          Target:
            URI: authz.example.com:9191
        AllowedHeaders:
        - Prefix: x-
        DisallowedHeaders:
        - Exact: authorization
          IgnoreCase: true
```

</CodeBlockConfig>

For the full list of configurable fields, including request and response header controls, refer to the [external authorization extension configuration reference](/consul/docs/reference/proxy/extensions/ext-authz).

Apply the resource to your cluster:

```shell-session
$ kubectl apply -f gateway-service-defaults.yaml
```

## Set the gateway-wide authorization behavior

External authorization is enabled by default for all routes attached to the `Gateway` resource. This step is optional and only necessary when you want to change that default.

To enable external authorization only for specific routes, add the `consul.hashicorp.com/ext-authz` annotation to the `Gateway` resource and set it to `disabled`, then enable it individually for the desired routes as shown in the next section.

<CodeBlockConfig filename="gateway.yaml">

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: Gateway
metadata:
  name: api-gateway
  annotations:
    consul.hashicorp.com/ext-authz: disabled
spec:
  gatewayClassName: consul
  listeners:
  - name: http-listener
    port: 8443
    protocol: HTTP
```

</CodeBlockConfig>

Apply the resource to your cluster:

```shell-session
$ kubectl apply -f gateway.yaml
```

## Override the authorization behavior for specific routes

You can override the gateway-wide default for individual routes in one of two ways.

### Use a RouteAuthFilter

Define a [`RouteAuthFilter`](/consul/docs/reference/k8s/api-gateway/routeauthfilter) and reference it from an `HTTPRoute` with an `extensionRef` filter. The filter's `spec.extAuthz.enabled` value takes precedence over the gateway-wide default.

<CodeBlockConfig filename="route-auth-filter.yaml">

```yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: RouteAuthFilter
metadata:
  name: admin-ext-authz
  namespace: default
spec:
  extAuthz:
    enabled: true
```

</CodeBlockConfig>

<CodeBlockConfig filename="http-route.yaml">

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: HTTPRoute
metadata:
  name: admin-route
spec:
  parentRefs:
  - name: api-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /admin
    filters:
    - type: ExtensionRef
      extensionRef:
        group: consul.hashicorp.com
        kind: RouteAuthFilter
        name: admin-ext-authz
    backendRefs:
    - name: admin
      port: 8080
```

</CodeBlockConfig>

Apply the resources to your cluster:

```shell-session
$ kubectl apply -f route-auth-filter.yaml
$ kubectl apply -f http-route.yaml
```

### Use an annotation

Alternatively, add the `consul.hashicorp.com/ext-authz` annotation directly to an `HTTPRoute` to override the authorization behavior for the routes it defines. Set the value to `enabled` or `disabled`.

<CodeBlockConfig filename="http-route-annotation.yaml">

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: HTTPRoute
metadata:
  name: admin-route
  annotations:
    consul.hashicorp.com/ext-authz: enabled
spec:
  parentRefs:
  - name: api-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /admin
    backendRefs:
    - name: admin
      port: 8080
```

</CodeBlockConfig>

Apply the resource to your cluster:

```shell-session
$ kubectl apply -f http-route-annotation.yaml
```

After Consul reconciles the resources, the API gateway calls the external authorization service according to the authorization behavior you configured before it forwards matching requests to upstream services.
