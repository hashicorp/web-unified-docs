# Resource upgrade

This topic describes what steps you need to take to complete the Kubernetes resource types upgrade, when a new version of Consul requires it.

## Overview

Kubernetes resources for Consul provide a way to configure and manage Consul on
Kubernetes clusters. Consul 2.0.0 provides new resource
types that offer enhanced functionality and improved compatibility
with OpenShift Container Platform (OCP) versions 4.19 and above. These new
resource types are part of the `consul.hashicorp.com` API group and are designed
to support the TCPRoute functionality in the Kubernetes platforms where Gateway CRDs are managed.

## Requirements

- Consul deployed in the Kubernetes.

## Upgrade scenarios

Kubernetes clusters not using the `TCPRoute` resource can, but do not need to, upgrade their API gateway resources to the new API group. This leaves two cases:

- OpenShift Container Platform (OCP) Kubernetes clusters that use TCPRoutes and want to upgrade their cluster from 4.18 to 4.19+.

    Moving to the new `consul.hashicorp.com` custom API resource group is required to ensure compatibility with OpenShift 4.19+.

- Any Kubernetes cluster which manages Gateway CRDs and uses TCPRoutes.

    Moving to the new `consul.hashicorp.com` custom API resource group is required to ensure compatibility.

## Upgrade steps

### OpenShift specific upgrade from 4.18 to 4.19+

1. Perform helm upgrade with below values in values.yaml.

<Tabs>

<Tab heading="No TCP Route - Use Standard Gateway">

<CodeBlockConfig filename="values.yaml">

```yaml
global:
    installK8sNetworkingCRDs: true
    generateManifests: true
    openshift:
      enabled: true
      crds:
        enableTcpRoute: false
```

</CodeBlockConfig>

</Tab>

<Tab heading="Require TCP Route - Use Consul Gateway">

<CodeBlockConfig filename="values.yaml">

```yaml
global:
    installK8sNetworkingCRDs: true
    generateManifests: true
    openshift:
      enabled: true
      crds:
        enableTcpRoute: true
        consulapi:
          enabled: true
```

</CodeBlockConfig>

</Tab>

</Tabs>

2. Verify the access to the application.
   - **Standard Gateway**
     - Verify the access to app with the gateway.

   - **Consul Gateway**
     - Users find 2 Gateways initially, try to access the app with both gateways.
     - Update your DNS (e.g. Route53) to point to Consul DNS.

3. Copy the generated manifests from PVC as backup.

4. Follow the official OpenShift documents to perform OCP platform upgrade which involves:
   - deletion of Gateway CRDs.
   - Approving adminAck and so on

5. Verify the access to the application after platform upgrade.
   - **Standard Gateway**
     - apply the manifests copied.
     - Once gateway is up, try to access the application.
    
   - **Consul Gateway**
     - Try to access the application.
     - if access to application is fails, perform helm upgrade with:

       ```yaml
       openshift:
          enabled: true
          upgradeTo419From418: false
          crds:
            enableTcpRoute: false
              consulapi:
              enabled: true
        ```

### Non-OCP upgrade flow : Platform upgrade involved.

Here we assume that current platform version does not manage Gateway CRDs and the upgraded platform version manages the Gateway CRDs.

1. Perform helm upgrade

<Tabs>

<Tab heading="No TCPRoute required - Standard Gateway">

<CodeBlockConfig filename="values.yaml">

```yaml
global:
    installK8sNetworkingCRDs: true
    generateManifests: true
    crds:
        enableTcpRoute: false
```

</CodeBlockConfig>

</Tab>

<Tab heading="TCPRoute required - Consul Gateway">

<CodeBlockConfig filename="values.yaml">

```yaml
global:
    installK8sNetworkingCRDs: true
    generateManifests: true
    crds:
        enableTcpRoute: true
        consulapi:
            enabled: true
```

</CodeBlockConfig>

</Tab>

</Tabs>

2. Verify access to the application
   - **Standard Gateway**
     - Verify the access to app with the gateway.
   - **Consul Gateway**
     - Users find 2 Gateways initially, try to access the app with both gateways.
     - Update your DNS (e.g. Route53) to point to Consul DNS.
     - Delete the Standard Gateway and its objects.

3. Copy the generated manifests.

4. Upgrade the platform following the official documentation of the platform.

5. Verify to access the application.
    - **Standard Gateway**
      - Verify the access to app with the gateway.
    - **Consul Gateway**
      - Verify the access to app with the gateway.
      - If not accessible, perform the helm upgrade 

        ```yaml
        global:
            installK8sNetworkingCRDs: false
            crds:
                enableTcpRoute: false
                consulapi:
                    enabled: true
        ```

To upgrade your Kubernetes resources for Consul, follow these steps:

1. Update your Consul Helm chart values. 

    This Helm upgrade generates manifests and applies the stable `v1` API
    version by replacing the `v1alpha` API version of the
    `gateway.networking.k8s.io` resources and then applies the manifests. 
    
    Add this configuration to your `values.yaml` file.

    ```yaml
    openshift:
        enabled: true
        upgradeTo419From418: true
        crds:
            enableTcpRoute: false
            consulapi:
                enabled: true
    ```

    If `openshift.crds.enableTcpRoute` is true, then install the TCPRoute CRD under the `gateway.networking.k8s.io`.
    If `openshift.crds.consulapi.enabled` is true, then install all the CRDs under the API group `consul.hashicorp.com`.

<Note>
 Make backups of all the manifests after the Consul upgrade or copy the manifests from PVC before proceeding to the OCP upgrade.

</Note>
