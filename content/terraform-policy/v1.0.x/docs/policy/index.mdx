# Terraform policy

Terraform policy is a streamlined policy framework with a familiar HCL-based syntax that integrates directly with Terraform providers, letting you seamlessly evaluate policies as part of your normal infrastructure workflows. Use Terraform policy to enforce policies across your organizations with HCP Terraform.

## Overview

Terraform policy introduces a declarative HashiCorp Configuration Language (HCL) based policy language that integrates directly with Terraform and its wide ecosystem of providers, letting you write policies that can reference any resource defined in a Terraform provider in a syntax familiar to Terraform practitioners.

In order to implement Terraform policy for your organization, you will:

1. Write policies for resources defined in any Terraform provider using Terraform policy's configuration language, based on HCL.
1. Test your policies locally with the `tfpolicy` CLI to ensure each policy targets the correct resources and behaves as you expect.
1. Commit your policies to a Version Control System (VCS) repository and connect it to HCP Terraform.
1. Organize policies into policy sets on HCP Terraform and target projects, workspaces, or your entire organization for evaluation.
1. Trigger runs on workspaces to have Terraform policy evaluate policies and report the results in HCP Terraform.

## Policy as code

Policy as code is the practice of writing code in a high-level language to define, test, and automate policy enforcement. By representing policies as code in text files, you can apply software development best practices to your policy enforcement, such as version control, testing, and automated deployments.

Policy as code provides a number of benefits, including:

- **Codification**: By representing policy logic as code using HCL, Terraform policy allows you to develop policies alongside your infrastructure configuration, and use the same workflows to develop, manage, and deploy your policies as you do with Terraform configuration.

- **Version Control**: Policies are stored as text files and managed by your version control system. This lets you take advantage of all the benefits of a modern VCS such as history, diffs, and pull requests.

- **Testing**: You can write tests along with your policies to help ensure that they work as you expect before you enforce them within your organization. Policy tests live alongside your policies, and become part of your change management process, helping you to avoid errors when you make changes to your policies.

- **Automation**: Policies provide guardrails for automated infrastructure deployment. As the number of automated systems grow, there is also a growing need to protect those automated systems from performing dangerous actions. Implementing policy as code allows you to include policy enforcement in your infrastructure management pipeleines.

## Evaluation and enforcement

HCP Terraform uses Terrafrom policy to evaluate your policies during Terraform runs. HCP Terraform will evaluate each policy as early as possible during each run, depending on when the data the policy refers to is available. You can also configure an enforcement level with each policy, depending on whether you want a policy violation to halt a run or produce a warning message.

### Evaluation stages

You can use Terraform policy to enforce policy at multiple points during a workspace run. Not all infrastructure data is available at the start of a run, so Terraform policy supports policy evaluation at three distinct stages of each Terraform run:

- **Pre-plan**: Provider and module policies that evaluate their meta-arguments are executed during init time, before Terraform installs the providers or modules. A policy violation prevents Terraform from downloading these providers and modules.

- **Plan time**: If all of the referenced attributes are available at plan time, Terraform policy evaluates policies against the Terraform plan before any infrastructure is changed. These policies are ideal for catching configuration errors before Terraform makes changes to your infrastructure.

- **Apply time**: Some attributes will not have determined values until Terraform applies your changes. Terraform marks these attributes as `"(known after apply)"` when it creates a plan. Apply-time evaluation allows policies to run after Terraform makes changes to your infrastructure and determines these attribute values. Since Terraform must apply the changes before determining if there is a policy violation, HCP Terraform will notify users of any non-complaint infrastructure after the Terraform apply is complete, and users must manually resolve the policy violation and redeploy the infrastructure.

### Enforcement levels

The enforcement model for Terraform policy allows you to designate an enforcement level for each policy depending on whether you want the policy to produce a warning message or block deployments, and whether users with the correct permissions can override the policy enforcement and continue the run in spite of a policy violation.

- **Advisory**: Advisory policies provide feedback to the user but do not stop the run. Use these policies to inform users about potential issues or to socialize policy standards before making them mandatory.

- **Mandatory-overridable**: If a mandatory-overridable policy fails at the pre-plan or plan phase, HCP Terraform will block the apply phase. HCP Terraform users with the `Manage policy overrides` permission can override these policy failures and enable the apply. Since Terraform cannot evaluate apply time policies until after it makes infrastructure changes, HCP Terraform will report these policies failures, but a user must resolve them and redeploy the infrastructure to return to compliance.

- **Mandatory**: If a mandatory policy fails at pre-plan or plan time, HCP Terraform will immediately halt the run and block the apply phase. Users cannot override mandatory policies. As with mandatory-overridable policies, HCP Terraform will report policy violations that occur after the apply phase, but a user must resolve them and redeploy the infrastructure to return to compliance.

## Organize policies in HCP Terraform

After you write your policies, test them, and commit them to a VCS repository, you will organize them into policy sets in HCP Terraform. Then you will configure those policy sets to apply to specific workspaces, projects, tags, or to your entire organization. HCP Terraform will automatically evaluate all policies in applicable policy sets whenever you plan and apply changes to a workspace, and report any errors to the end user. Refer to [HCP Terraform policy enforcement overview](/terraform/cloud-docs/workspaces/policy-enforcement) for more information.
