# Policy test blocks

Like policy files, Terraform policy's test syntax is based on HCL, and supports a number of block types that you can use when you write tests for your policies.

Each policy test file can include:

- An optional ['policytest {}' block](#the-policytest--block) to configure which policies Terraform policy will test.
- One or more [`provider {}`](#the-provider--block), [`resource {}`](#the-resource--block), [`data {}`](#the-data--block), and [`module {}`](#the-module--block) blocks to define mock providers, resources, data sources, and modules for Terraform policy to test against your policies.
- Any number of [`locals {}`](#the-locals--block) blocks to define local values.

## The `policytest {}` block

The `policytest` block defines the policies that Terraform policy will test.

You can specify a top level block named `policytest` that lists which policy files Terraform policy will test your resources against. The `targets` attribute is a list of strings representing policy files or directories containing policy files for Terraform policy to test against the mock providers, resources, data sources, and modules you define in the test file.

```hcl
policytest {
  targets = [
    "path/to/single_policy.policy.hcl",
    "path/containing/multiple_policies"
  ]
}
```

If you do not specify targets in your `policytest` block, Terraform policy will evaluate the policy files in the path you provide when you run your tests.

## Mock providers, resources, data sources, and modules

To test your policies, you will author mock providers, resources, data sources, and modules in your policy test files. When you run your tests with Terraform policy, it will evaluate your policies against these mock objects to ensure they work as expected. As a best practice, we recommend that you include at least one passing and one failing test case for each policy you wish to test.

### The `provider {}` block

To test a provider policy, add a mock provider block to your test file. Use the `meta` attribute to set meta-attributes, including the provider source and version. Use the `attrs` block to set values for any attributes defined by the provider itself.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
provider "aws" "example" {
  meta = {
    source  = "hashicorp/aws"
    version = "5.0.0"
  }
  
  attrs = {
    region = "us-east-1"
  }
}
```

</CodeBlockConfig>

### The `resource {}` block

To test a resource policy, add a mock resource block to your test file. Use the `meta` attribute to set meta-attributes for the resource, if required. Use the `attrs` attribute to set values for any resource attributes defined by the provider. To set the value of block arguments, you must assign values to a matching named attribute with the `=` operator. Refer to [Replace block arguments](#replace-resource-block-arguments) for more information. If you need to mock resource or data source relationships to test your resource policy, refer to [Test resource and data source relationships](#test-resource-and-data-source-relationships) for more information.

### The `data {}` block

Terraform data sources do not represent infrastructure directly managed by Terraform, so you will not directly write policies for them. However, in many Terraform configurations, data sources serve as the foundation for resource relationships, providing IDs or configurations fetched from existing infrastructure. You may need to mock these data sources in your policy tests to ensure that your policies correctly verify resources or modules that rely on this data. To do so, add a mock data source block to your test file. Use the `meta` attribute to set meta-attributes, if required. Use the `attrs` block to set values for any attributes returned by the data source that your resource or module policies rely on.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
data "aws_vpc" "corporate_network" {
  attrs = {
    id         = "vpc-12345678"
    cidr_block = "10.0.0.0/16"
  }
}
```

</CodeBlockConfig>

Unlike resources, which might have a proposed state in a plan, data sources in Terraform are read-only. In your tests, data blocks serve as a static source of attributes that resource or module mocks can consume to match the behavior of the Terraform workspace you will enforce your policies on.

### The `module {}` block

To test a module policy, add a mock module block to your test file. Use the `meta` attribute to set meta-attributes, including the module source and version. Use the `attrs` block to set values for any attributes defined by the module itself.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
module "app.terraform.io/my-org/vpc/aws" "example" {
  meta = {
    source  = "app.terraform.io/my-org/vpc/aws"
    version = "6.6.1"
  }

  attrs = {
    name = "example-vpc"
    cidr = "10.0.0.0/16"

    azs             = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
    private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
    public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]
  }
}
```

</CodeBlockConfig>

## Replace block arguments

Terraform resource and provider arguments use various types, including nested block arguments. For most argument types, you will assign a value of the matching type within the `attrs` argument. However, Terraform policy does not support including blocks within the `attrs` object. In order to mock providers and resources that use nested blocks, replace the block with a named attribute.

For example, the following Terraform configuration manages an instance of the `azurerm_linux_virtual_machine` resource type, which uses nested block arguments to configure an SSH key, disk, and source image:

<CodeBlockConfig filename="example.tf">

```hcl
resource "azurerm_linux_virtual_machine" "example" {
  name                = "public-vm"
  resource_group_name = azurerm_resource_group.rg-public.name
  
  admin_ssh_key {
    username   = "azureuser"
    public_key = tls_private_key.vm_ssh_key.public_key_openssh
  }
  
  os_disk {
    caching              = "ReadWrite"
    storage_account_type = "Standard_LRS"
    name                 = "public-os-disk"
  }
  
  source_image_reference {
    publisher = "Canonical"
    offer     = "0001-com-ubuntu-server-jammy"
    sku       = "22_04-lts-gen2"
    version   = "latest"
  }
}
```

</CodeBlockConfig>

In Terraform configuration, a block argument is defined by the name of the argument, followed by a block represented by `{ }`.

When you write mock providers or resources for your Terraform policy tests, you can assign object or list values in the `attrs` argument to represent the block body. For example, the following resource block mocks an `azurerm_linux_virtual_machine` resource including the `admin_ssh_key` block by assigning an object representing the block to the `admin_ssh_key` attribute.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
resource "azurerm_linux_virtual_machine" "example" {
  attrs = {
    admin_ssh_key = {
      username   = "azureuser"
      public_key = tls_private_key.vm_ssh_key.public_key_openssh
    }

##...Other attributes

  }
}
```

</CodeBlockConfig>

### Nested block types

The type of each argument, including nested blocks, is defined by the provider's schema. There are three types of nested blocks available in the Terraform provider schema:

| Block Type | Use Case | Terraform policy attribute value |
|------------|----------|----------------------------|
| Single nested | Single structure of attributes/blocks | Assign an object value to an attribute with the same name as the block. |
| Set nested | Unordered, unique collection of structures of attributes/blocks | Assign a list of object values. |
| List nested | Ordered collection of structures of attributes/blocks | Assign a list of object values. |

The syntax for using nested blocks in Terraform policy tests depends on the underlying provider schema of that block argument. For single nested blocks, you will assign an object value representing the block. For set and list nested blocks, you will assign a list of object values representing the block or blocks. Refer to the provider documentation to determine the type of block required by a given provider or resource.

For example, the `service_account` argument of the `google_compute_instance` resource is a set nested block. The following example Terraform configuration defines a compute instance with two service accounts:

<CodeBlockConfig filename="example.tf">

```hcl
resource "google_compute_instance" "example" {
  name         = "my-instance"
  machine_type = "n2-standard-2"
  zone         = "us-central1-c"
  
  service_account {
    email  = google_service_account.cloud_platform.email
    scopes = ["cloud-platform"]
  }

  service_account {
    email  = google_service_account.monitoring.email
    scopes = ["monitoring"]
  }
}
```

</CodeBlockConfig>

To assign values to one or more `service_account` blocks in your Terraform policy tests, assign a list of objects to the `service_account` attribute.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
resource "google_compute_instance" "example" {
  attrs = {
    name         = "my-instance"
    machine_type = "n2-standard-2"
    zone         = "us-central1-c"
    
    service_account = [
      {
        email  = google_service_account.cloud_platform.email
        scopes = ["cloud-platform"]
      },
      {
        email  = google_service_account.monitoring.email
        scopes = ["monitoring"]
      }
    ]
  }
}
```

</CodeBlockConfig>

While set nested and list nested block arguments differ in implementation within Terraform providers, you will assign values to them the same way in Terraform policy tests.

## Test resource and data source relationships

Policies often need to validate relationships between different resources and data sources. For example, ensuring that an AWS CloudTrail resource is correctly linked to an encrypted S3 bucket. You can model these dependencies by allowing mock resources and data sources to reference one another, just as they would in a real Terraform configuration.

### Resource relationships

When testing policies that use the [`core::getresources()`](/terraform/policy/write/functions#the-getresources-function) function, Terraform policy builds a graph of the mock resources defined in your test file. You can create a relationship between two mock resources by referencing an attribute of one resource within the attrs block of another.

You may mock resources or data sources to establish resource relationships that your policies will test, without needing to test the policies against those particular resources directly. Set the `skip` attribute to `true` on resource or data source blocks you do not want Terraform policy to evaluate against your policies. This tells Terraform policy to include the resource in the dependency graph,making its attributes available for lookup, but to bypass any policy evaluations for the mocked resource.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
resource "aws_s3_bucket_acl" "example" {
  skip = true
  
  attrs = {
    bucket = "example-bucket"
    acl    = "private"
  }
}

resource "aws_cloudtrail" "example" {
  attrs = {
    name           = "example-log"
    s3_bucket_name = aws_s3_bucket_acl.example.bucket
    enable_logging = true
  }
}
```

## Test block attributes

Terraform policy supports the following optional attributes within provider, resource, data source, and module blocks in test files:

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `attrs` | Sets the attributes for the provider, resource, data source, or module. Terraform policy will set these attributes on your mock when it tests your policies. You can set values for any attribute that the provider supports for the given provider or resource blocks, or for any input variable defined in the given module. | Object | `{bucket = "my-bucket", acl = "private"}` |
| `expect_failure` | Expect the mock to fail the matching policies. A block with this attribute set to `true` passes instead of fails if the resource fails the matching policies. Use to test that your policy fails in expected cases. | Boolean | `true`, `false` |
| `meta` | Provides data to configure meta-arguments for the mock provider, resource, data source, or module. | Object | `{address = "aws_s3_bucket.example"}` |
| `operation` | Set the operation to test. | String | `"create"`, `"update"`, `"delete"` |
| `prior_attrs` | The value of the attributes when the operation is set to `update` or `delete`. If `prior_attrs` is set while `operation` is not set, Terraform policy assumes the operation is an "update". | Object | `{bucket = "old-bucket"}` |
| `skip` | Set to true to have Terraform policy skip evaluating the given mock against your policies. Use for mocks that only establish resource or data source relationships. | Boolean | `true`, `false` |

<Note>

Terraform policy will error if it encounters invalid policies or tests, such as policies that request resources or data sources that have no data configured, or those that cannot be parsed due to invalid syntax. Terraform policy will raise these errors even if the `expect_failure` or `skip` attributes are set to `true`.

</Note>

## Testing operation-based policies

When a resource or module policy uses the `operations` attribute to scope itself to a specific Terraform operation, your test cases must indicate which action each mock resource represents.

The testing framework provides two optional attributes within mock resource blocks for this purpose: `operation` and `prior_attrs`.

### The `operation` attribute

Use the operation attribute to declare which planned action a mock resource represents with values: "create", "update" or "delete". If you do not specify a value for the operation attribute, Terraform policy will default to "create", unless you set the `prior_attrs` attribute, in which case it will default to "update".

### The `prior_attrs` attribute

Use the prior_attrs attribute to populate the mock data for the prior_attrs variables used by the policy evaluation context.

<CodeBlockConfig filename="example.policytest.hcl">

```hcl
resource "aws_s3_bucket" "tag_unchanged" {
  operation = "update"
  
  prior_attrs = {
    bucket = "demo-bucket"
    tags = {
      Owner = "platform-team"
    }
  }
  
  attrs = {
    bucket = "demo-bucket"
    tags = {
      Owner = "platform-team"
    }
  }
}

```

</CodeBlockConfig>
