# Attributes and meta-arguments

Within any policy block (`resource_policy`, `provider_policy`, or `module_policy`), you have access to data about your configuration, infrastructure state, and environment through attributes and meta-arguments. Attributes provide data about your configuration and infrastructure, while meta-arguments provide data about your environment from Terraform itself.

## Attributes

Attributes correspond to the provider, resource, or module attributes defined in Terraform. When it evaluates your policies, Terraform policy assigns values to these attributes based on your configuration, plan, or state, depending on when those values are available. For provider attributes, Terraform can compute them by reading your configuration pre-plan, so it does so. For provider and module policies, the values of some attributes will be available at plan time. Other values depend on the successful completion of the apply step, and are marked as `(known after apply)` in Terraform plan output. Terraform will read the value from your state after applying the change.

Attribute values can be complex types like maps and lists, which you can reference in your policies via dot notation (For example: `attrs.tags.environment`). Use the `attrs` attribute to reference the target state of a provider, resource, or module, and the `prior_attrs` attribute to reference to previous state, if any.

## Meta-arguments

These are built-in metadata fields provided by the Terraform policy engine. You can use meta-arguments to provide context in your policies, such as where a resource lives in the module tree or what version of a provider is being used. Provider and module meta-arguments such as "source" and "version" are available before the plan is complete, so Terraform policy will evaluate policies that refer to these meta-arguments at the pre-plan stage.

## Available meta-arguments

### The `filter` meta-argument

Terraform policy will apply policies to the providers, resources, or modules which match the first label in the policy's block, filtered by the value of the optional `filter` meta-argument, if provided.

The value of the filter meta attribute must evaluate to a boolean value. Use comparison expressions to compare against attributes and other values. For example:

```hcl
resource_policy "aws_instance" "example" {
  filter = attrs.tags.env == "testing"

# ...
}
```

### The `enforcement_level` meta-argument

Terraform policy enforce your policies at the "advisory" level by default. Use the `enforcement_level` meta-argument to set the enforcement level for the given policy.

The value of the enforcement_level meta-argumment must be a string, either: "advisory", "mandatory", or "mandatory-overridable".

```hcl
resource_policy "aws_instance" "example" {
  enforcement_level = "mandatory-overridable"

# ...
}
```

### The `operations` meta-argument

By default, Terraform policy will enforce your policy during Terraform operations as soon as all of the attributes referred to in the policy block are availble. You can restrict Terraform policy to only enforce your policy at certain phases of a Terraform operation with the optional `operations` meta-argument.

Depending on the operation being evaluated, the value of the `attrs` and `prior_attrs` attributes will contain different values:

| Operation | attrs.<name> (Post-Change) | prior_attrs.<name> (Pre-Change) |
|-----------|----------------------------|----------------------------------|
| create    | Target State               | (unavailable/null)               |
| update    | Target State               | Current State                    |
| delete    | Target State (all resource attributes are null) | Current State |

### Policy type meta-arguments

In addition to the meta-arguments above, each policy block supports meta-arguments specific to that type of policy.

#### Provider policy meta-arguments

The following meta-arguments are available in `provider_policy` blocks:

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `address` | Address of the provider within Terraform. | String | `"provider["registry.terraform.io/hashicorp/aws"]"` |
| `alias` | Alias given to the provider. | String | `"us-west"`, `"production"` |
| `name` | The local name of the provider. | String | `"aws"`, `"azurerm"` |
| `namespace` | The provider's registry namespace. | String | `"hashicorp"`, `"example-corp"` |
| `source` | The full source of the provider. | String | `"hashicorp/aws"`, `"hashicorp/azurerm"` |
| `type` | The type of provider. | String | `"aws"`, `"azurerm"` |
| `version` | The resolved version of the provider. | String | `"5.0.0"`, `"3.75.0"` |

#### Resource policy meta-arguments

The following meta-arguments are available in `resource_policy` blocks:

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `module_path` | The module path the resource belongs to. | String | `"."`, `"my-module"` |
| `operation` | The operation being performed. | String | `"create"`, `"update"`, `"destroy"` |
| `provider_type` | The name of the provider of the resource. | String | `"aws"`, `"azurerm"` |
| `resource_type` | The type of the resource. This will be the first label of the matching resource block. | String | `"aws_s3_bucket"`, `"azurerm_managed_disk"` |

#### Module policy meta-arguments

The following meta-arguments are available in `module_policy` blocks:

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `address` | Address of the module within Terraform. | String | `"module.vpc"`, `"module.networking.module.subnets"` |
| `source` | The full source of the module. | String | `"app.terraform.io/my-org/vpc/aws"`, `"./modules/networking"` |
| `version` | The resolved version of the module. | String | `"2.1.0"`, `"1.0.0"` |
