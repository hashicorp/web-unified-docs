# Policy blocks

Terraform policy's syntax is based on HCL, and supports a number of block types that you can use when you write your policies. Unlike Terraform, Terraform policy evaluates each policy file in a given directory separately. We recommend that you include sets of policies that you want to enforce together in each policy file, and related policies in separate files in the same directory structure.

## Policy block syntax

Terraform policy block syntax is based on HCL, and similar to Terraform's syntax.

```hcl
<block_type> "<label>" ["<name>"] {
  <attribute_name> = <attribute_value>
  ...

  <block_name> {
    <block_attr_name> = <block_attr_value>
    ...
  }
}
```

Each policy file can include:

- An optional ['policy {}' block](#the-policy--block) to configure Terraform policy itself.
- One or more [`provider_policy {}`](#the-provider-policy--block), [`resource_policy {}`](#the-resource-policy--block), and [`module_policy {}`](#the-module-policy--block) blocks to define your policies.
- Any number of [`input {}`](/terraform/policy/write/inputs-and-locals#the-input--block) blocks to define input variables.
- Any number of [`locals {}`](/terraform/policy/write/inputs-and-locals#the-locals--block) blocks to define local values.

Terraform policy will evaluate all polices in a policy file as a unit, as early in the Terraform run as possible, when all values that the given policy block refers to are available.

## The `policy {}` block

Use the policy block to configure how Terraform policy evaluates your policies, including the version of Terraform that Terraform policy will use to evaluate your policies.

The `policy` block supports the optional `terraform_config` attribute, which defines settings which are specific to Terraform. The only currently supported attribute is `required_version`, which validates that the current Terraform CLI version fulfills the specified version constraint. The `required_version` attribute uses the same version constraint syntax as Terraform itself. Refer to [Version constraints - Version constraint syntax](/terraform/language/expressions/version-constraints#version-constraint-syntax) for more information.

The `policy` block also supports the optional `plugins` attribute, which allows you to refer to binary plugins that provide custom functions you can use in your policies. Refer to [Terraform policy - Policy plugins](/terraform/policy/write/plugins) for more information.

```hcl
policy {
  terraform_config {
    required_version = "> 1.16.0"
  }
  plugins {
    example = {
      source = "../plugins/bin/example"
    }
  }
}
```

The above example requires that the Terraform version used to evaluate these policies be greater than "1.16.0".

## Policy blocks

Terraform policy supports three types of block that allow you to write policies for your Terraform providers, resources, and modules.

Each policy block requires two labels. The first label will match the target provider, resource, or module, and the second gives the policy a unique name within the same policy file. Optionally, you can use `*` for all or part the first label to match multiple providers, resources, or modules in your Terraform configuration.

By default, Terraform policy will apply a given policy to all of the providers, resources, or modules that match the given label. You can filter the matching providers, resources, or modules with the [`filter` meta-argument](/terraform/policy/write/attributes#the-filter-meta-argument).

By default, Terraform policy will treat your policies as advisory. You can set the enforcement level with the optional [`enforcement_level` meta-argument](/terraform/policy/write/attributes#the-enforcement_level-meta-argument). There are three enforcement levels available: "advisory", "mandatory-overridable" and "mandatory".

By default, Terraform policy will enforce your policy as early in a Terraform operation as possible. You can specify the phases during which Terraform policy will evaluate your policy with the optional [`operations` meta-argument](/terraform/policy/write/attributes#the-operations-meta-argument).

Each policy block must contain one or more `enforce {}` blocks that define a condition that must evaluate to `true` for the policy to pass. Refer to [The `enforce {}` block](#the-enforce--block) for more informaiton.

### Policy block types

Use the corresponding Terraform policy block type to enforce policies for your providers, resources, or modules.

#### The `provider_policy {}` block

Use the provider_policy block to define policies for Terraform providers. The first label of the block identifies the provider name, such as `aws`, `google`, or `azurerm`, and the second label is the policy name. You can use a wildcard (*) for the provider name to apply a single policy to every provider defined in the configuration.

Within the provider policy block, the `attrs` object provides access to the arguments defined in the provider block, such as region settings or account identifiers. The `meta` object provides access to Terraform-defined informationa bout the provider, including the source and version. If you use provider aliasing in your Terraform configurations, you can specify which provider alias to evaluate with a filter on the `alias` meta-attribute in the `provider_policy {}` block.

```hcl
provider_policy "aws" "standard_configuration" {
  locals {
    version_constraint = core::semverconstraint(meta.version, ">= 5.0.0")
  }

  enforce {
    condition     = attrs.region == "us-east-1"
    error_message = "All AWS resources must be deployed in the us-east-1 region."
  }

  enforce {
    condition     = meta.source == "hashicorp/aws"
    error_message = "Only the official HashiCorp AWS provider is permitted."
  }

  enforce {
    condition     = local.version_constraint
    error_message = "The AWS provider version must be at least 5.0.0."
  }
}
```

The example above ensures that only authorized provider binaries can be used to manage your infrastructure. It also uses the core library function `semverconstraint()` to compare the provider version.

#### The `resource_policy {}` block

Use the resource_policy block to define policies for Terraform resources. The first label of the block identifies the resource name, such as `aws_instance`, `google_compute_instance`, or `azurerm_windows_virtual_machine`, and the second label is the policy name. You can use a wildcard (*) for all or part of the resource name to apply a single policy to every resource defined in the configuration, or every resource starting with a given prefix. Optionally, use the `filter` meta-argument to filter resources based on their attributes.

```hcl
resource_policy "aws_instance" "testing" {
  operation         = ["create", "update"]
  enforcement_level = "mandatory"
  filter            = attrs.tags.env == "testing"
  
  enforce {
    condition     = attrs.instance_type == "t3.micro"
    error_message = "Only t3 micro instances are allowed in testing environments. Please consider changing the instance_type to t3.micro."
  }
}
```

The example above targets "aws_instance" resources during create and update operations, with a mandatory enforcement level. Instead of matching all AWS resources, this policy filters AWS instance resources by the value their `tags.env`. This policy ensures that any AWS instance tagged as being part of the "testing" environment can only use "t3.micro" instance types.

#### The `module_policy {}` block

Use the module_policy block to define policies for your workspaces' Terraform modules. Module policies allow you to set guardrails at the abstraction layer where infrastructure is often packaged and shared. The first label of the block specifies the module source—the value of the `source` attribute for the target module blocks. The second label is a unique name for the policy itself. You can use a wildcard (*) for the module source to apply the policy to every module in the workspace. As with other policy blocks, the second label is the name of the policy, allowing you to include multiple policy blocks for each module source.

```hcl
locals {
  approved_sources = [
    "./modules/networking",
    "git::github.com/org/terraform-aws-vpc"
  ]
}

# Global check for all modules in the workspace
module_policy "*" "version_and_source_check" {
  enforce {
    condition     = core::contains(local.approved_sources, meta.source)
    error_message = "The module source ${meta.source} is not in the approved list."
  }
  
  enforce {
    condition     = core::semverconstraint(meta.version, ">= 1.0.0")
    error_message = "Module version must be at least 1.0.0."
  }
}

# Specific check for the VPC module source
module_policy "git::github.com/org/terraform-aws-vpc" "specific_version_check" {
  enforce {
    condition     = core::semverconstraint(meta.version, ">= 2.0.0")
    error_message = "The official VPC module must be at version 2.0.0 or higher."
  }
}
```

The meta attribute name provides details about the module's identity, such as its version and its logical address within the configuration. By using both, you can enforce standards that ensure only approved versions of specific modules are used, and that they are configured according to organizational best practices.

Module policies do not support regular attributes at this time.

The core::contains refers to the contains() function available in Terraform. See other functions available when you write policies here.

### Policy block arguments and blocks

Terraform policy supports the following optional meta-arguments within policy blocks:

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `operation` | Set the operation to enforce the policy against. If not specified, Terraform policy will enforce your policy during "create" and "update" operations. | String | `"create"`, `"update"`, `"delete"` |
| `enforcement_level` | The enforcement level for the policy. Defaults to "advisory". | String | `"advisory"`, `"mandatory"`, `"mandatory-overridable"` |
| `filter` | Skip evaluating the given mock against your policies. Use for mocks that only establish resource or data source relationships. Defaults to `false`. | Boolean | `true`, `false` |

#### The `enforce {}` block

The enforce block allows you to write conditions for Terraform policy to test against the underlying provider, resource, or module. Each enforce block includes the condition for Terraform policy to test, and an error message.

```hcl
module_policy "git::github.com/org/terraform-aws-vpc" "example" {
  enforce {
    condition     = core::semverconstraint(meta.version, ">= 2.0.0")
    error_message = "The official VPC module must be at version 2.0.0 or higher."
    info_message  = "VPC module version ${meta.version}."
  }
}
```

You can include one or more enforce blocks in each policy block. If any of the enforce blocks' conditions fail during evaluation, then Terraform policy will fail the policy and return the corresponding error messages.

You can also include an optional `info_message` within an enforce block to print information irrespective of whether the policy passes or fails. Use an `info_message` to print out information retrieved by a policy during evaluation, such as the result of an http request or evaluating complex conditions. These messages can clutter diagnostics, so we recommend using them sparingly.
