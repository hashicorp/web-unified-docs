# The `resource_policy {}` block

Use the `resource_policy {}` block to define policies for Terraform resources. The first label of the block identifies the resource type, such as `aws_instance`, `google_compute_instance`, or `azurerm_windows_virtual_machine`, and the second label is the policy name. You can use a wildcard (*) for all or part of the resource name to apply a policy to every resource defined in the configuration, or every resource starting with a given prefix.

## Configuration model

```hcl
resource_policy "<resource_type>" "<policy_name>" {
  operation         = "<operation>"
  enforcement_level = "<level>"
  filter            = <boolean_expression>

  locals {
    name = <value>
  }

  enforce {
    condition     = <boolean_expression>
    error_message = "<message>"
    info_message  = "<message>"
  }
}
```

## Complete configuration example

```hcl
resource_policy "aws_instance" "testing" {
  operation         = ["create", "update"]
  enforcement_level = "mandatory"
  filter            = attrs.tags.env == "testing"
  
  enforce {
    condition     = attrs.instance_type == "t3.micro"
    error_message = "Only t3.micro instances are allowed in testing environments. Please consider changing the instance_type to t3.micro."
  }
  
  enforce {
    condition     = attrs.monitoring == true
    error_message = "Monitoring must be enabled for all instances."
  }
}
```

## Specification

### Labels

| Label | Required | Description |
|-------|----------|-------------|
| First label | Yes | The resource type (For example: `aws_instance`, `google_compute_instance`). Use `*` to match all resources, or `aws_*` to match all AWS resources. |
| Second label | Yes | A unique name for this policy within the policy file. |

### Arguments

Resource policy blocks support the following arguments.

| Argument | Required | Type | Default | Description |
|----------|----------|------|---------|-------------|
| `operation` | No | String or List of Strings | `["create", "update"]` | The operations during which to enforce this policy. Valid values: `"create"`, `"update"`, `"delete"`. |
| `enforcement_level` | No | String | `"advisory"` | The enforcement level for the policy. Valid values: `"advisory"`, `"mandatory"`, `"mandatory-overridable"`. |
| `filter` | No | Boolean expression | - | Filter which resources the policy applies to based on their attributes. |

### Attributes

Within the `resource_policy {}` block, you have access to the following attributes.

| Object | Description |
|--------|-------------|
| `attrs` | Provides access to the resource attributes. |
| `prior_attrs` | Provides access to the prior state of resource attributes. |
| `meta` | Provides access to Terraform-defined information about the resource. |

Depending on the operation being evaluated, the value of the `attrs` and `prior_attrs` attributes will contain different values as described in the following table.

| Operation | attrs.<name> | prior_attrs.<name> |
|-----------|----------------------------|----------------------------------|
| create    | Target State               | (unavailable/null)               |
| update    | Target State               | Current State                    |
| delete    | Target State (all resource attributes are null) | Current State |

Each Terraform provider defines the attributes available for each resource it supports. Refer to the provider documentation for a list of supported attributes.

#### `meta` object attributes

The following meta-arguments are available in `resource_policy` blocks.

| Name | Description | Type | Example |
|------|-------------|------|---------|
| `meta.module_path` | The module path the resource belongs to. | String | `"."`, `"my-module"` |
| `meta.operation` | The operation being performed. | String | `"create"`, `"update"`, `"destroy"` |
| `meta.provider_type` | The name of the provider of the resource. | String | `"aws"`, `"azurerm"` |
| `meta.resource_type` | The type of the resource. This will be the first label of the matching resource block. | String | `"aws_s3_bucket"`, `"azurerm_managed_disk"` |

### The `enforce {}` block

Each `resource_policy {}` block must contain one or more `enforce {}` blocks that define conditions that must evaluate to `true` for the policy to pass.

#### Arguments

Enforce blocks support the following arguments.

| Argument | Required | Type | Description |
|----------|----------|------|-------------|
| `condition` | Yes | Boolean expression | A boolean expression that must evaluate to `true` for the policy to pass. |
| `error_message` | Yes | String | The error message to display when the condition evaluates to `false`. |
| `info_message` | No | String | An informational message to display regardless of whether the policy passes or fails. |

You can include one or more `enforce {}` blocks in each policy block. If any of the enforce blocks' conditions fail during evaluation, then Terraform policy will fail the policy and return the corresponding error messages.

### The `locals {}` block

Each `resource_policy {}` block may contain `locals {}` blocks that define values that are local to the policy. These local values override `locals {}` blocks defined at the top level of the policy file. Refer to [The `locals {}` block](/terraform/policy/reference/policy/locals) for more information.

## Examples

### Enforce instance type in testing environment

```hcl
resource_policy "aws_instance" "testing" {
  operation         = ["create", "update"]
  enforcement_level = "mandatory"
  filter            = attrs.tags.env == "testing"
  
  enforce {
    condition     = attrs.instance_type == "t3.micro"
    error_message = "Only t3.micro instances are allowed in testing environments."
  }
}
```

This example targets `aws_instance` resources during create and update operations, with a mandatory enforcement level. Instead of matching all AWS instances, this policy filters AWS instance resources by the value of their `tags.env` attribute.

### Enforce encryption for EBS volumes

```hcl
resource_policy "aws_ebs_volume" "encryption_required" {
  enforcement_level = "mandatory"

  enforce {
    condition     = attrs.encrypted == true
    error_message = "All EBS volumes must be encrypted."
  }
}
```

### Use wildcard to match multiple resource types

```hcl
resource_policy "aws_*" "require_tags" {
  enforcement_level = "mandatory-overridable"

  enforce {
    condition     = attrs.tags != null && core::length(core::keys(attrs.tags)) > 0
    error_message = "All AWS resources must have at least one tag."
  }
}
```

This example uses a wildcard to apply the policy to all AWS resources.

### Enforce security group rules

```hcl
resource_policy "aws_security_group" "no_public_ingress" {
  enforcement_level = "mandatory"

  locals {
    has_public_ingress = core::contains(
      [for rule in attrs.ingress : rule.cidr_blocks],
      ["0.0.0.0/0"]
    )
  }

  enforce {
    condition     = !local.has_public_ingress
    error_message = "Security groups must not allow ingress from 0.0.0.0/0."
  }
}
```

### Multiple enforce blocks

```hcl
resource_policy "aws_s3_bucket" "security_requirements" {
  enforcement_level = "mandatory"

  enforce {
    condition     = attrs.versioning[0].enabled == true
    error_message = "S3 bucket versioning must be enabled."
  }

  enforce {
    condition     = attrs.server_side_encryption_configuration != null
    error_message = "S3 bucket must have server-side encryption configured."
  }

  enforce {
    condition     = attrs.acl != "public-read" && attrs.acl != "public-read-write"
    error_message = "S3 bucket must not have public ACL."
  }
}
```
