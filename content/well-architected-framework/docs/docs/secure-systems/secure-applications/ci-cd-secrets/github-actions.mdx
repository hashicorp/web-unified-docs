# GitHub Actions integration with Vault

GitHub Actions workflows can authenticate with Vault to securely retrieve secrets for CI/CD deployments. Instead of storing credentials in GitHub Secrets, workflows authenticate to Vault at runtime using OpenID Connect (OIDC) — a token-based authentication standard — and retrieve secrets on demand.

## Why integrate GitHub Actions with Vault

Integrating GitHub Actions with Vault addresses the following critical security and operational challenges in workflow automation:

**Eliminate secret sprawl across repositories:** Vault centralizes secret storage so workflows retrieve credentials on-demand from a single source. When a credential rotates, a single Vault update propagates to all workflows instead of requiring updates across every repository.

**Enable automatic secret rotation:** Vault automates credential rotation on a configurable schedule and distributes updated credentials without requiring workflow configuration changes. Teams no longer need to track which repositories hold which credentials or coordinate rotation across projects.

**Meet compliance and audit requirements:** Vault provides comprehensive audit logs that track every secret access with workflow identity, timestamp, and source IP, supporting regulatory compliance frameworks that require detailed access trails. Audit logs are tamper-evident and can be shipped to centralized SIEM platforms.

**Provide dynamic credentials:** Vault generates temporary credentials for each workflow run with a configurable TTL that automatically expires. Each workflow run receives unique credentials traceable through audit logs, minimizing the blast radius of any single credential compromise.

## Choose an integration approach

GitHub Actions supports several approaches for integrating with Vault. Choose from the following approaches based on your security requirements and workflow complexity:

| Approach | Setup Complexity | Security Posture | Credential Storage | Best For |
|----------|-----------------|------------------|-------------------|----------|
| [GitHub OIDC provider](/vault/docs/auth/jwt/oidc-providers) | Medium | Excellent | None | Teams wanting strongest security without credentials |
| [Vault Secrets Sync](/vault/docs/sync/github) (Enterprise) | Low | Good | GitHub Secrets | Teams wanting minimal workflow changes |
| [HashiCorp Vault GitHub Action](https://github.com/hashicorp/vault-action) | Medium | Excellent | None (with OIDC) | Teams needing flexible auth methods and dynamic secrets |

## Best practices

The following best practices help you secure Vault integration with GitHub Actions:

- **Use OIDC over long-lived tokens:** Prefer GitHub's OIDC provider over storing Vault tokens or AppRole (application role) credentials in GitHub Secrets. OIDC generates a short-lived JSON Web Token (JWT) per workflow run with no stored credentials to manage or rotate. Only use AppRole when OIDC is unavailable.

- **Restrict access with bound claims:** Configure JWT roles with `bound_claims` using GitHub's OIDC token fields `repository_owner`, `repository`, and `ref` to limit which repos and branches can authenticate. Set `bound_claims_type` to `glob` for wildcard pattern matching on claim values. Read the [JWT/OIDC auth method documentation](/vault/docs/auth/jwt) for configuration details.

- **Enforce least-privilege policies:** Grant each workflow read access only to the exact Vault paths it needs. Create separate policies for test, deploy, and release workflows rather than sharing a single broad policy.

- **Use dynamic secrets for infrastructure credentials:** Retrieve database and cloud credentials from Vault's secrets engines (`aws/creds`, `database/creds`) rather than storing static credentials. Dynamic secrets expire automatically based on their configured time-to-live (TTL), eliminating the need for manual rotation.
