# Google Cloud Build

Google Cloud Build runs builds using a Google Cloud service account that provides native authentication to Vault through the [GCP auth method](/vault/docs/auth/gcp). Instead of storing credentials in Google Secret Manager or build substitution variables, Cloud Build steps authenticate to Vault at runtime and retrieve secrets on demand.

## Why integrate Google Cloud Build with Vault

Integrating Google Cloud Build with Vault addresses the following critical security and operational challenges in build automation:

**Eliminate secret sprawl across Google Cloud services:** Teams often duplicate secrets across Google Secret Manager, Cloud Build substitution variables, and build trigger configurations, making rotation difficult and increasing the risk of exposure. Vault centralizes secret storage so builds retrieve credentials on-demand from a single source, reducing duplication across GCP projects.

**Enable automatic secret rotation:** Vault automates credential rotation and distributes updated credentials without requiring changes to Cloud Build configurations or `cloudbuild.yaml` files, reducing operational overhead and maintaining security posture.

**Meet compliance and audit requirements:** Vault provides comprehensive audit logs that track every secret access with build identity and timestamp, supporting regulatory compliance frameworks that require detailed access trails beyond what Cloud Audit Logs provides for Secret Manager access.

**Provide dynamic credentials:** Vault generates temporary credentials for each build that automatically expire, minimizing exposure windows compared to long-lived static credentials stored in Secret Manager or environment variables.

## Choose an integration approach

Google Cloud Build supports several approaches for integrating with Vault. Choose from the following approaches based on your security requirements and infrastructure:

| Approach | Setup Complexity | Security Posture | Credential Storage | Best For |
|----------|-----------------|------------------|-------------------|----------|
| [GCP IAM auth](/vault/docs/auth/gcp) | Low | Excellent | None | Cloud Build using default or custom service accounts |
| [Vault Secrets Sync](/vault/docs/sync/gcpsm) (Enterprise) | Low | Good | GCP Secret Manager | Teams wanting minimal build configuration changes |
| [AppRole](/vault/docs/auth/approle) | Medium | Good | GCP Secret Manager | Multi-cloud builds or environments without GCP IAM |

## Best practices

The following best practices help you secure Vault integration with Google Cloud Build:

- **Use GCP IAM auth with bound service accounts:** Configure the GCP auth method with `bound_service_accounts` set to the specific Cloud Build service account email. The IAM auth type verifies identity using signed JWTs from GCP metadata without transmitting credentials. Read the [GCP auth method documentation](/vault/docs/auth/gcp) for role configuration details.

- **Set short token TTLs:** Configure Vault token TTL values that match your build execution duration — typically 15-60 minutes for most Cloud Build projects. Short TTLs reduce the risk window if a token is leaked during build execution.

- **Enforce least-privilege policies per trigger:** Create separate Vault roles and policies for each Cloud Build trigger or project, with each role restricted to specific secret paths. Grant read access only to the exact secrets each build needs, avoiding wildcard permissions.

- **Use dynamic secrets for cloud and database credentials:** Retrieve database and cloud credentials from Vault's dynamic secrets engines rather than static KV secrets. Dynamic secrets expire automatically after the build completes, eliminating manual rotation.
