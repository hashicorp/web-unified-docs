# Dynamic and static secrets in CI/CD

CI/CD pipelines require access to secrets like database credentials, API keys, and cloud provider credentials. You can use two types of secrets in your pipelines: dynamic secrets that are generated on-demand and automatically expire, or static secrets that are manually created and remain until explicitly changed. Understanding the differences helps you choose the right approach for your security requirements and operational constraints.

## Comparing secret types

| Aspect | Dynamic secrets | Static secrets (key-value/KV) |
|--------|----------------|----------------|
| **Creation** | Generated on-demand | Manually created and stored |
| **Lifecycle** | Automatic expiration/revocation | Manual rotation/revocation |
| **Lifespan** | Short-lived (minutes to hours) | Long-lived (until changed) |
| **Uniqueness** | Unique per request | Shared across users/systems |
| **Risk** | Lower | Higher |
| **Setup complexity** | Higher | Lower |
| **Operational overhead** | Lower | Higher |

## Understanding dynamic and static roles

Vault supports two different approaches for managing credentials across secrets engines: dynamic roles and static roles. Understanding the difference helps you choose the right pattern for your CI/CD pipelines.

**Dynamic roles** generate unique credentials on-demand such as a username and password, cloud access key, or service account, and automatically revoke them when the lease expires. Each pipeline run gets its own dedicated credentials, which keeps audit trails precise and prevents credential reuse.

**Static roles** map to a pre-existing account, such as a database user or service account, and rotate only the credentials on a schedule. The identity stays fixed; only the credentials change.

## When to use dynamic secrets

Use dynamic secrets when the external service supports the following types of programmatic credential generation:

**Cloud providers:** AWS, Azure, and GCP secrets engines generate temporary access credentials with automatic expiration. Use for pipelines that provision or interact with cloud resources.

**Databases:** The database secrets engine creates unique credentials for each pipeline run and automatically revokes them on expiration. Use for applications that need database access during deployment or testing.

**PKI certificates:** The Public Key Infrastructure (PKI) secrets engine issues short-lived certificates for mutual TLS (mTLS) authentication. Use for service-to-service communication in microservices deployments.

## When to use static secrets

Use static secrets when dynamic generation is not possible or practical. The following scenarios are common use cases for static secrets:

**Third-party API keys:** Use for external services that don't support programmatic credential generation. Store static API keys in Vault's KV secrets engine with access control and audit logging.

**Shared certificates:** Use for third-party SSL/TLS certificates you cannot generate programmatically. The KV secrets engine provides versioning and centralized storage. For certificates you control, prefer Vault's PKI secrets engine to issue short-lived certificates dynamically.

**Legacy systems:** Use for applications that cannot integrate with dynamic secrets engines. Retrieve static credentials from Vault instead of storing them in code or configuration files.

**Configuration values:** Use for non-sensitive configuration data that requires centralized management and audit trails.

## Best practices for dynamic secrets in CI/CD

The following best practices help you optimize security and operations when using dynamic secrets in CI/CD pipelines:

- **Set appropriate TTLs based on pipeline duration:** Configure `default_ttl` to match your typical pipeline execution time and `max_ttl` to cap the absolute maximum duration. Time to live (TTL) controls how long Vault credentials remain valid. For a 30-minute pipeline, set `default_ttl="1h"` to provide buffer time and `max_ttl="2h"` to prevent indefinite extensions.

- **Implement least-privilege roles:** Create separate Vault roles for different access levels rather than granting broad permissions. Define a read-only role for test pipelines that only need to verify data, and a separate read-write role for deployment pipelines that modify resources. Grant only the specific permissions each pipeline needs.

- **Revoke leases early when pipelines complete:** Use `vault lease revoke $LEASE_ID` at the end of successful pipeline runs to delete credentials immediately rather than waiting for TTL expiration. This reduces the exposure window from hours to minutes. Implement this in a cleanup step that runs even if previous steps fail.

- **Handle credential retrieval failures gracefully:** Wrap Vault API calls in error handling that fails the pipeline if credentials cannot be retrieved. Check for empty responses, invalid JSON, or connection errors before attempting to use credentials. Log errors with enough detail for debugging but avoid exposing credential values in logs.

- **Validate credentials before use:** Test retrieved credentials before proceeding with deployments. A simple connectivity or API check helps catch configuration errors, network issues, or permission problems early in the pipeline rather than discovering them during critical deployment steps.

- **Monitor lease creation and expiration rates:** Track how many leases your pipelines create per hour and monitor for unusual spikes that might indicate compromised credentials or misconfigured pipelines. Set up alerts for lease exhaustion to prevent service disruption when too many leases are created simultaneously.

- **Use connection pooling carefully with dynamic credentials:** Dynamic credentials that expire during application runtime can break connection pools. For long-running applications, either use static secrets with rotation or implement connection pool refresh logic that obtains new credentials before the lease expires. For short-lived CI/CD pipelines, this is typically not a concern.

## Performance considerations

The following considerations affect pipeline performance when using dynamic secrets:

**Dynamic secret latency:** Each dynamic secret request requires Vault to generate credentials from the external service, adding latency to pipeline execution. Use appropriate TTLs to balance security with performance.

**Caching strategies:** Use [Vault Agent caching](/vault/docs/agent-and-proxy/agent/caching) or [Vault Proxy](/vault/docs/agent-and-proxy/proxy) to reduce direct Vault API calls from high-frequency pipelines. Caching is especially effective for static secrets.

**Rate limiting:** Configure [rate limit quotas](/vault/docs/configuration/create-rate-limit-quota) to protect your Vault cluster from excessive load when running many pipelines in parallel.
