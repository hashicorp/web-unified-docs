# Argo CD

Argo CD is a Kubernetes-native GitOps continuous delivery tool that synchronizes application state from Git repositories to Kubernetes clusters. Integrating Vault with Argo CD keeps secrets out of Git while maintaining GitOps principles through the [Kubernetes auth method](/vault/docs/auth/kubernetes) and [Vault Secrets Operator](/vault/docs/platform/k8s/vso).

## Why integrate Argo CD with Vault

Integrating Argo CD with Vault addresses the following critical security and operational challenges in GitOps workflows:

**Prevent secret exposure in Git and Argo CD:** Secrets stored directly in Git manifests create security risks because Argo CD caches generated manifests in plaintext in Redis and exposes them via the repo-server API. Vault keeps secrets out of Git entirely, with only non-sensitive Custom Resource references committed to your repository.

**Simplify secret rotation workflows:** Rotating secrets stored in Git requires encrypting new values, committing changes, and triggering Argo CD sync operations. Vault rotates secrets independently of Git, updating Kubernetes Secrets without Argo CD sync intervention while maintaining audit trails.

**Eliminate per-cluster encryption key management:** Solutions like Sealed Secrets and SOPS require managing encryption keys in each Kubernetes cluster. Vault centralizes secret management with unified authentication and access control across clusters.

**Enable granular audit logging:** Kubernetes does not provide granular audit logging for Secret reads by default. Vault provides comprehensive audit logs that track every secret access with application identity and timestamp, supporting compliance and security investigations.

## Choose an integration approach

Several approaches exist for integrating Vault with Argo CD and Kubernetes. Choose from the following approaches based on your security requirements and workflow complexity:

| Approach | Setup Complexity | Key Feature | Best For |
|----------|-----------------|-------------|----------|
| [Vault Secrets Operator](/vault/docs/platform/k8s/vso) | Medium | Kubernetes-native CRDs, lowest Vault API load | GitOps workflows where secrets sync independently of Argo CD |
| [Vault Secrets Store CSI provider](/vault/docs/platform/k8s/csi) | Medium | Volume-mounted secrets without a sidecar | Applications that consume secrets as files or environment variables |
| [Argo CD Vault Plugin](https://argocd-vault-plugin.readthedocs.io/) | Medium | Inline manifest secret references at sync time | Teams wanting secret placeholders directly in GitOps manifests |
| [External Secrets Operator](https://external-secrets.io/) | Medium | Multi-backend secret synchronization | Organizations managing secrets from Vault and other backends |
| [Vault Agent sidecar](/vault/docs/platform/k8s/injector) | High | Go-template rendering, widest auth method support | Applications requiring templated secret files and maximum auth flexibility |

## Best practices

The following best practices help you secure and operationalize Vault integration with Argo CD:

- **Use Kubernetes namespaces for isolation:** Deploy applications in separate namespaces to create security boundaries. Configure Vault Kubernetes auth method roles to bind service accounts to specific namespaces, ensuring applications can only authenticate from their designated namespace.

- **Create dedicated Vault roles per application:** Define separate Vault roles for each application or service deployed by Argo CD, binding each role to the application's specific service account and namespace combination.

- **Implement least-privilege policies per application:** Write Vault policies that grant each application read access only to the specific secret paths it needs, avoiding wildcard permissions or broad path access.

- **Use dynamic secrets where possible:** Prefer dynamic secrets over static secrets for database credentials, cloud provider credentials, and other infrastructure access. Dynamic secrets provide automatic rotation and time-limited credentials without manual rotation workflows.
