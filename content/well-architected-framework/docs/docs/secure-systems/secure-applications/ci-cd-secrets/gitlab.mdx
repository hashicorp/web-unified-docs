# GitLab CI/CD integration with Vault

GitLab CI/CD pipelines can authenticate with Vault using JSON Web Tokens (JWT) to securely retrieve secrets at runtime. Instead of storing credentials in GitLab CI/CD variables, pipelines authenticate to Vault on each run and pull secrets on demand from Vault's secrets engines.

GitLab generates [ID tokens](https://docs.gitlab.com/ci/secrets/id_token_authentication/) for each CI/CD job, which Vault verifies using the [JWT auth method](/vault/docs/auth/jwt). After authentication, pipelines can retrieve static secrets from the [KV secrets engine](/vault/docs/secrets/kv) or dynamic credentials from engines such as the [AWS secrets engine](/vault/docs/secrets/aws).

## Why integrate GitLab with Vault

Integrating GitLab CI/CD with Vault addresses the following critical security and operational challenges in pipeline automation:

**Eliminate credential duplication across projects:** Vault centralizes secret storage so pipelines retrieve credentials on demand from a single source. When a credential rotates, a single Vault update propagates to all pipelines instead of requiring updates in every project's CI/CD variables.

**Prevent credential exposure in pipeline logs:** Vault's dynamic secrets and secure retrieval methods reduce the risk of credentials appearing in build output. Logs are frequently shipped to centralized observability platforms, where plaintext secrets become accessible to anyone with log query access.

**Enable short-lived dynamic credentials:** Vault generates dynamic credentials per pipeline run with configurable time-to-live (TTL) values. A compromised credential expires automatically, and each pipeline run receives unique credentials traceable through audit logs.

**Enforce fine-grained access control:** Vault policies control access to specific secret paths based on JWT claims such as `project_id`, `ref`, and `ref_protected`. Administrators can restrict production secrets to protected branches and specific projects, preventing unauthorized pipelines from retrieving sensitive credentials.

## Choose an integration approach

GitLab supports several approaches for integrating with Vault. Choose from the following approaches based on your GitLab tier and security requirements:

| Approach | GitLab Tier | Setup Complexity | Security Posture | Best For |
|----------|------------|-----------------|------------------|----------|
| [`secrets:vault` keyword](https://docs.gitlab.com/ci/secrets/hashicorp_vault/) | Premium, Ultimate | Low | Excellent | Teams wanting native, declarative secret retrieval |
| [Manual ID token authentication](https://docs.gitlab.com/ci/secrets/hashicorp_vault/#manual-id-token-authentication) | Free, Premium, Ultimate | Medium | Excellent | Teams on Free tier or needing custom retrieval logic |
| [Vault CLI in scripts](https://docs.gitlab.com/ci/secrets/hashicorp_vault/#manual-id-token-authentication) | Free, Premium, Ultimate | Medium | Good | Teams needing dynamic secrets or multi-step workflows |

## Retrieve static secrets

The `secrets:vault` keyword provides declarative secret retrieval in your `.gitlab-ci.yml` file. GitLab Runner handles authentication and injects secrets as environment variables.

In the following example, the pipeline authenticates to Vault with an ID token and retrieves a database password from the KV secrets engine.

```yaml title="Declarative secret retrieval"
deploy_to_production:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  secrets:
    DATABASE_PASSWORD:
      vault: production/db/password@ops
      token: $VAULT_ID_TOKEN
      file: false
  script:
    - deploy --db-password "$DATABASE_PASSWORD"
```

The `file: false` option stores the secret value directly in the environment variable. Without the option, GitLab writes the secret to a temporary file and sets the variable to the file path.

For teams on GitLab Free tier or those needing more control over the authentication flow, use manual ID token authentication with the Vault CLI. The following example authenticates with Vault's JWT auth method and retrieves the same database password through a CLI command.

```yaml title="Manual ID token authentication"
deploy_to_production:
  image: hashicorp/vault:latest
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  variables:
    VAULT_ADDR: https://vault.example.com:8200
  script:
    - export VAULT_TOKEN="$(vault write -field=token auth/jwt/login role=myproject-production jwt=$VAULT_ID_TOKEN)"
    - export DATABASE_PASSWORD="$(vault kv get -field=password secret/myproject/production/db)"
    - deploy --db-password "$DATABASE_PASSWORD"
```

The manual approach authenticates to the `auth/jwt/login` endpoint, receives a short-lived Vault token, and uses the token to read secrets. The token inherits the policies attached to the Vault role, which control which secret paths the pipeline can access.

## Retrieve dynamic secrets

Vault secrets engines generate dynamic credentials on demand. After authenticating with an ID token, pipelines can request short-lived credentials from engines such as the [AWS secrets engine](/vault/docs/secrets/aws) or [database secrets engine](/vault/docs/secrets/databases).

The following example retrieves temporary AWS credentials from Vault's AWS secrets engine. The credentials expire automatically based on the TTL configured in Vault.

```yaml title="Dynamic AWS credentials"
deploy_to_aws:
  image: hashicorp/vault:latest
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  variables:
    VAULT_ADDR: https://vault.example.com:8200
  script:
    - export VAULT_TOKEN="$(vault write -field=token auth/jwt/login role=myproject-aws jwt=$VAULT_ID_TOKEN)"
    - export AWS_CREDS="$(vault read aws/creds/my-role -format=json)"
    - export AWS_ACCESS_KEY_ID=$(echo "${AWS_CREDS}" | jq -r .data.access_key)
    - export AWS_SECRET_ACCESS_KEY=$(echo "${AWS_CREDS}" | jq -r .data.secret_key)
    - export AWS_SESSION_TOKEN=$(echo "${AWS_CREDS}" | jq -r .data.security_token)
    - aws s3 sync ./build s3://my-deployment-bucket
```

Read the [AWS secrets engine documentation](/vault/docs/secrets/aws) for configuration details.

## Supported secrets engines

GitLab Runner natively supports the following Vault secrets engines through the [`secrets:engine:name`](https://docs.gitlab.com/ci/yaml/#secretsvault) keyword. Review the [GitLab secret engines](https://docs.gitlab.com/ci/secrets/hashicorp_vault/#secrets-engines) documentation for the most up-to-date list of supported engines.

| Secrets Engine | `engine:name` Value | Min Runner Version | Default | Details |
|---------------|--------------------|--------------------|---------|---------|
| [KV version 2](/vault/docs/secrets/kv/kv-v2) | `kv-v2` | 13.4 | Yes | Default engine when no type is specified |
| [KV version 1](/vault/docs/secrets/kv/kv-v1) | `kv-v1` or `generic` | 13.4 | No | Use `generic` for KV v1 paths |
| [AWS](/vault/docs/secrets/aws) | `generic` | 16.11 | No | Retrieves dynamic AWS credentials |
| [Artifactory](https://developer.hashicorp.com/vault/docs/secrets/artifactory) | `generic` | 16.11 | No | Retrieves dynamic JFrog Artifactory access tokens |

## Best practices

The following best practices help you secure Vault integration with GitLab CI/CD:

- **Restrict production secrets to protected branches:** Set `"ref_protected": "true"` in Vault role bound claims to ensure only [protected branches](https://docs.gitlab.com/user/project/repository/branches/protected/) can access production secrets. Combined with GitLab's branch protection rules, Vault prevents unauthorized pipelines from retrieving sensitive credentials.

- **Enforce least-privilege policies:** Grant each Vault role read access only to the specific secret paths the pipeline needs. Create separate policies and roles for staging and production environments rather than sharing a single broad policy.

- **Use dynamic secrets for cloud and database credentials:** Retrieve cloud provider and database credentials from Vault's secrets engines rather than storing static credentials. Dynamic secrets expire automatically based on their configured TTL, eliminating the need for manual rotation.

- **Set short token TTLs:** Configure `token_explicit_max_ttl` on Vault roles to limit how long the Vault token remains valid. A TTL of 60 to 300 seconds is sufficient for most pipeline jobs and limits the window of exposure if a token is compromised.
