# Tekton

[Tekton](https://tekton.dev/) is a Kubernetes-native CI/CD framework that runs pipeline tasks as pods. Because Tekton executes on Kubernetes, it integrates with Vault using the same Kubernetes-native approaches as other cluster workloads — the [Kubernetes auth method](/vault/docs/auth/kubernetes) and [Vault Secrets Operator](/vault/docs/platform/k8s/vso). Vault keeps secrets out of pipeline YAML and Kubernetes Secrets manifests, providing centralized management with audit trails.

## Why integrate Tekton with Vault

Integrating Tekton with Vault addresses the following critical security and operational challenges in Kubernetes-native CI/CD pipelines:

**Prevent secret exposure in pipeline definitions:** Tekton Tasks and Pipelines stored in Git repositories may reference Kubernetes Secrets by name, but the Secret values themselves should not be committed to version control. Vault keeps sensitive values out of Git entirely, with only non-sensitive references in pipeline definitions.

**Simplify secret rotation across pipelines:** Rotating secrets stored in Kubernetes Secrets requires updating each Secret object and restarting affected pods. Vault Secrets Operator automatically syncs updated secrets to Kubernetes without manual intervention, ensuring Tekton Tasks receive current credentials.

**Eliminate per-namespace secret management overhead:** Managing Kubernetes Secrets across multiple namespaces for different pipeline workloads creates operational overhead. Vault centralizes secret management with unified authentication and access control across namespaces and clusters.

**Enable granular audit logging:** Kubernetes does not provide granular audit logging for Secret reads by default. Vault provides comprehensive audit logs that track every secret access with pipeline identity and timestamp, supporting compliance and security investigations.

## Choose an integration approach

Several approaches exist for integrating Vault with Tekton Pipelines. Choose from the following approaches based on your security requirements and workflow complexity:

| Approach | Setup Complexity | Key Feature | Best For |
|----------|-----------------|-------------|----------|
| [Vault Secrets Operator](/vault/docs/platform/k8s/vso) | Medium | Kubernetes-native CRDs, lowest Vault API load | Pipelines consuming secrets as environment variables or mounted files |
| [Vault Secrets Store CSI provider](/vault/docs/platform/k8s/csi) | Medium | Volume-mounted secrets without a sidecar | Tasks that consume secrets as files in specific Steps |
| [Vault Agent sidecar](/vault/docs/platform/k8s/injector) | High | Go-template rendering, widest auth method support | Tasks requiring templated secret files and maximum auth flexibility |
| [Direct Vault CLI in Tasks](/vault/docs/commands) | Low | No cluster-level components required | Teams wanting explicit Vault interaction in pipeline Steps |

## Best practices

The following best practices help you secure and operationalize Vault integration with Tekton:

- **Use Kubernetes namespaces for pipeline isolation:** Deploy pipeline workloads in separate namespaces to create security boundaries. Configure Vault Kubernetes auth method roles to bind service accounts to specific namespaces, ensuring Tasks can only authenticate from their designated namespace.

- **Create dedicated Vault roles per pipeline:** Define separate Vault roles for each Tekton Pipeline or Task, binding each role to the pipeline's specific service account and namespace combination. Avoid sharing Vault roles across unrelated pipelines.

- **Implement least-privilege policies per pipeline:** Write Vault policies that grant each pipeline read access only to the specific secret paths it needs, avoiding wildcard permissions or broad path access.

- **Use dynamic secrets for infrastructure credentials:** Prefer dynamic secrets over static secrets for database credentials, cloud provider credentials, and other infrastructure access. Dynamic secrets provide automatic rotation and time-limited credentials without manual rotation workflows.
