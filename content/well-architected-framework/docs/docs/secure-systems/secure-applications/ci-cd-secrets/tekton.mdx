# Secure Tekton CI/CD secrets with HashiCorp Vault

Tekton runs natively on Kubernetes, which changes the threat model compared
to hosted CI/CD platforms. Pipelines execute as pods with service accounts —
identities that already exist in the Kubernetes cluster and that Vault can
verify without distributing long-lived credentials to pipeline workloads. This
makes Kubernetes auth the natural fit for most Tekton deployments because it
reduces bootstrap secret distribution and lets you bind Vault access directly
to Kubernetes workload identity.

<VideoEmbed url="https://www.youtube.com/watch?v=EltMXWz35RY"/>

## Architecture

The pipeline-to-Vault authentication flow follows the same high-level pattern
regardless of auth method, but the verification step differs by method:

1. The workload presents an identity to Vault
1. Vault connects to the workload's platform
1. The platform validates the workload's identity
1. The platform returns the workload identity information to Vault
1. Vault maps the identity to a role and issues a token based on the policies attached to that role

## Authentication options

Because Tekton runs natively on Kubernetes, use the Kubernetes auth for
pipelines running inside the same cluster as Vault or a cluster with network
access to Vault. 

- The pod's service account token is already available
- The identity is verifiable by the Kubernetes API
- No long-lived, static credentials needed for pipeline pod
- Each pipeline can have its own service account, enabling per-pipeline secret scoping with no shared
  credentials

Depending on your use case, there are additional auth methods that may be
appropriate for Tekton pipelines in your environment.

| Auth method | Security posture | Credential lifecycle | Best suited for |
|---|---|---|---|
| Kubernetes | Highest for in-cluster | Short-lived projected token on modern Kubernetes; legacy patterns vary | Pipelines running in same cluster or trusted cluster with Vault access |
| JWT / OIDC | High; identity federates across trust boundaries | Token issued per job or workload, typically short-lived | Multi-cluster, external runners, or where OIDC-compatible workload identity is in use |
| TLS Certificate | High; mutual authentication | Certificate with defined expiry; requires rotation process | Long-lived pipeline infrastructure (e.g., dedicated build agents) where mTLS is already part of the security posture |
| AppRole | Moderate; requires secure secret ID delivery | Secret ID is single-use with response wrapping; RoleID is long-lived | Pipelines in environments where neither K8s nor JWT is available; legacy infrastructure integration |

<Tabs>
<Tab heading="Kubernetes">

Vault validates the pipeline pod's projected service account token against the
Kubernetes API.

### Role configuration

The role definition enforces which Kubernetes namespaces and service
accounts Vault allows to authenticate, making this the primary control against
one pipeline impersonating another.

```hcl
# Vault role — binds to a specific service account and namespace
# This is the critical security control: without bound_service_account_names
# and bound_service_account_namespaces, any pod in the cluster can authenticate
vault write auth/kubernetes/role/tekton-build \
  bound_service_account_names=tekton-pipeline-sa \
  bound_service_account_namespaces=tekton-pipelines \
  policies=tekton-build-policy \
  ttl=1h
```

</Tab>
<Tab heading="JWT / OIDC">

JWT auth is the right choice when pipelines run outside the cluster boundary or
when external systems other than the local Kubernetes API manage the workload
identity.

### Role configuration

The security-critical configuration here is workload-specific role
constraints. `bound_claims` is the most flexible option, but you can also use
constraints such as `bound_audiences` and `bound_subject`. Without these
restrictions, a broader set of issuer-valid JWTs may authenticate than you
intend.

```hcl
# Vault JWT role with bound_claims
# Without bound_claims this role accepts any JWT from the issuer
vault write auth/jwt/role/tekton-deploy \
  role_type=jwt \
  bound_audiences="https://vault.example.com" \
  bound_claims='{"spiffe.io/trust-domain": "example.com", "sub": "spiffe://example.com/ns/tekton-pipelines/sa/deploy-sa"}' \
  user_claim=sub \
  policies=tekton-deploy-policy \
  ttl=15m
```

This example assumes the JWT contains SPIFFE-style claims. If your identity
provider issues different claims, bind the role to the claim names and values
that uniquely identify the pipeline workload in your environment.

</Tab>
<Tab heading="TLS Certificate">

TLS cert auth is appropriate when pipelines run on dedicated, long-lived build
agents rather than ephemeral pods — for example, external build workers or
build farm nodes integrated with Tekton where the agent infrastructure itself has a managed
certificate lifecycle. Vault validates the client certificate presented during
the TLS handshake. 

### Role configuration

Trust in the issuing CA is the primary control in cert auth. CN or SAN
restrictions then narrow authentication to the specific certificate subjects
you expect. In practice, this means you allow certificates issued by a specific
CA and optionally restrict login to a defined naming pattern.

```hcl
# Vault TLS cert role — restrict to certificates from the internal CA
# with a CN matching the build agent naming pattern
vault write auth/cert/certs/tekton-agent \
  display_name=tekton-agent \
  certificate=@/path/to/internal-ca.crt \
  allowed_common_names="tekton-agent-*.infra.example.com" \
  policies=tekton-agent-policy \
  ttl=4h
```

</Tab>
<Tab heading="AppRole">

AppRole is appropriate when neither Kubernetes nor JWT auth is available — for
example, pipelines in air-gapped environments or integrations with legacy
infrastructure that can't participate in token-based identity federation.

### Role configuration

The critical operational concern with AppRole is secure initial delivery of the
secret_id. Always use response wrapping: Vault wraps the secret_id in a
single-use token so the credential can only be unwrapped once.

Without response wrapping, AppRole delivery in pipelines often degrades into
manually distributed bootstrap secrets, even if the secret ID itself is short-
lived. That weakens the security model and increases operational risk.

```hcl
# Vault AppRole role — short TTL on secret_id, single use
vault write auth/approle/role/tekton-pipeline \
  role_name=tekton-pipeline \
  secret_id_ttl=10m \
  secret_id_num_uses=1 \
  token_ttl=1h \
  token_max_ttl=2h \
  policies=tekton-pipeline-policy
```

</Tab>
</Tabs>

## Policy

The Vault policy for any role should follow least privilege. A pipeline that
deploys to production should not be able to read staging secrets, and vice
versa.

<VideoEmbed url="https://www.youtube.com/watch?v=Mmyvybxm30w"/>

An example Vault policy for a production deployment pipeline might look like this:

```hcl
# Vault policy: scope to exactly the paths this pipeline needs
path "secret/data/ci/production/db" {
  capabilities = ["read"]
}

# Deny access to everything else implicitly
```

## Security considerations

There are several security considerations you need to align with your
organizational risk tolerance and operational constraints when designing your
CI/CD secrets management strategy.

For Tekton, the primary consideration is how to scope and manage Kubernetes
namespaces and service accounts.  Every distinct pipeline workload should have
its own service account with its own Vault role. A shared service account across
pipelines is a lateral movement path — if one pipeline is compromised, it can
access secrets scoped to any other pipeline sharing that identity.

For a list of additional security considerations, refer to the [CI/CD security
considerations](/well-architected-framework/secure-systems/secure-applications/ci-cd-secrets/security-considerations)
page.

## Integrate HCP Vault Radar

Before migrating to Vault-based secrets management, Vault Radar can scan
existing Tekton pipeline definitions and connected repositories for
credentials stored as hardcoded values. This is particularly important in
organizations that have accumulated years of workflows with credentials stored
across multiple environments, and repositories.

After migration, Vault Radar can monitor your source code repositories
continuously for new secret introductions — catching cases where developers
hardcode credentials that should be going through Vault.

Refer to the [HCP Vault Radar quick start
tutorials](/hcp/tutorials/get-started-hcp-vault-radar) to learn about HCP Vault
Radar.
