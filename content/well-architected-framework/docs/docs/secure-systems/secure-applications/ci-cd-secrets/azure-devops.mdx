# Azure DevOps

Azure DevOps Pipelines can authenticate with Vault to securely retrieve secrets for CI/CD workflows. Instead of storing credentials in Azure DevOps variable groups or pipeline variables, pipelines authenticate to Vault at runtime and retrieve secrets on demand.

## Why integrate Azure DevOps with Vault

Integrating Azure DevOps with Vault addresses the following critical security and operational challenges in pipeline automation:

**Eliminate secret sprawl across projects:** Vault centralizes secret storage so pipelines retrieve credentials on-demand from a single source, reducing the need to duplicate secrets across multiple Azure DevOps projects, variable groups, and service connections. Centralized storage simplifies rotation and reduces the risk of exposure.

**Enable automatic credential rotation:** Static secrets stored in Azure DevOps variable groups remain valid indefinitely, creating a larger attack surface compared to short-lived dynamic credentials that expire automatically. Vault automates credential rotation and distributes updated credentials without requiring pipeline configuration changes.

**Meet compliance and audit requirements:** Organizations need detailed audit trails showing which pipelines accessed which secrets and when, but Azure DevOps native logging may not provide the granularity required for regulatory compliance frameworks like SOC 2, PCI DSS, and HIPAA. Vault provides comprehensive audit logs that track every secret access with pipeline identity and timestamp.

**Provide dynamic credentials:** Manually managing Azure service principal credentials and database passwords creates operational overhead and security risk when credentials are shared across pipeline stages. Vault generates temporary credentials for each pipeline run that automatically expire, limiting the exposure window if credentials leak through logs, error messages, or compromised pipeline agents.

## Choose an integration approach

Azure DevOps supports several approaches for integrating with Vault. Choose from the following approaches based on your security requirements and Azure infrastructure:

| Approach | Setup Complexity | Security Posture | Credential Storage | Best For |
|----------|-----------------|------------------|--------------------|----------|
| [OIDC (Azure AD)](/vault/docs/auth/jwt) | Medium-High | Excellent | None | Teams with Azure AD and workload identity federation |
| [Azure Managed Identity](/vault/docs/auth/azure) | Medium | Excellent | None | Self-hosted agents running on Azure VMs |
| [AppRole](/vault/docs/auth/approle) | Low | Good | Azure DevOps variable groups | Microsoft-hosted agents or environments without Azure AD |

## Best practices

The following best practices help you secure Vault integration with Azure DevOps Pipelines:

- **Configure bound claims for Azure AD tokens:** When using OIDC authentication, configure Vault JWT auth roles with `bound_claims` that restrict authentication to specific Azure AD tenant IDs, subscription IDs, or service principal object IDs. Read the [Azure AD OIDC provider documentation](/vault/docs/auth/jwt/oidc-providers/azuread) for Azure-specific bound claims configuration.

- **Set short token TTLs:** Configure Vault token TTL values that match your pipeline execution duration — typically 15-30 minutes for most Azure Pipelines. Short TTLs reduce the risk window if tokens are compromised. Longer TTLs may be needed for multi-stage deployment pipelines.

- **Enforce least-privilege policies per pipeline:** Create Vault policies that grant each pipeline or project the minimum permissions required. Grant read access only to the specific secret paths each pipeline needs, avoiding wildcard permissions.

- **Use dynamic secrets for Azure and database credentials:** Retrieve Azure service principal and database credentials from Vault's dynamic secrets engines rather than static KV secrets. Dynamic secrets expire automatically after the pipeline completes, eliminating manual rotation.
