# Automate policy enforcement with policy as code

Manual policy enforcement is subjective and can miss violations. As your organization grows, ensuring that deployments follow standards is challenging. Operations like provisioning infrastructure, managing secrets, and deploying services require adherence to specific standards and organizational requirements.

Policy as code addresses these challenges. When you write policy checks in declarative languages like Sentinel or OPA and store them in a version control system, you can apply software development best practices to policy management. Use practices such as peer review, automated testing, and automated deployment to bring consistency, automation, and auditability to your policy management. 

HashiCorp's policy as code product, Sentinel, controls what policies get into production. It integrates with HCP Terraform to validate configurations before provisioning resources. Sentinel catches non-compliant infrastructure before going to production and provides audit trails for your organization.

## Why policy as code?

Policy as code addresses the following challenges:

- **Prevent configuration drift:** Manual policy enforcement can lead to inconsistencies and configuration drift. Policy as code ensures that policies are consistently applied across all environments, reducing the risk of misconfigurations.
- **Reduce security risk:** By automating policy enforcement, you can catch security violations before they reach production, reducing the risk of breaches and vulnerabilities.
- **Control cloud costs:** Policy as code can enforce cost-saving measures, such as restricting the use of expensive instance types or ensuring resources are tagged for cost allocation, helping to manage and optimize cloud spending.
- **Standardize controls:** By defining policies as code, you can standardize controls across your organization, ensuring that all teams adhere to the same security and compliance standards through automated testing and peer review processes.
- **Scale policy management:** As your organization grows, manually enforcing policies becomes impractical. Policy as code allows you to manage policies at scale across multiple teams and projects.

## How policy as code works

When you provision infrastructure or manage secrets, you want to ensure they follow certain behaviors. By managing policy as code, you can unify the process of maintaining and implementing policies across your entire infrastructure stack. 

When a user attempts to provision infrastructure, HashiCorp Sentinel evaluates the defined policies before allowing the action to proceed. If the proposed changes violate any policies, Sentinel prevents the action and provides feedback on the violation.

You write policies in a declarative language, such as Sentinel, which specifies the desired state and rules for your infrastructure and applications.

A common use case for policy as code is to control cloud costs. For example, you can write a policy that restricts the creation of certain instance types that are more expensive, ensuring that your infrastructure stays within budget.

```hcl hideClipboard
import "tfplan"

main = rule {
  all tfplan.resources.aws_instance as _, instances {
    all instances as _, r {
      r.applied.instance_type in ["t2.micro", "t2.small", "t2.medium"]
     }
  }
}
```

This policy checks all AWS EC2 instances and ensures that only the specified instance types are provisioned. If a user attempts to create an instance type outside of the allowed list, the policy will prevent the action.

You store policy code in a version control system (VCS) like Git, enabling team collaboration, change tracking, and rollback capabilities. Your teams review policies through pull requests, ensuring that changes are reviewed and approved before deployment. This process promotes accountability and transparency in policy management.

Using tools like the [Sentinel testing framework](https://developer.hashicorp.com/sentinel/docs/writing/testing), you can write unit tests for your policies to validate their behavior against expected outcomes. These tests help ensure that policies function as intended and automatically catch issues early in the development process.

You can integrate policy unit tests with [CI/CD pipelines](https://www.hashicorp.com/en/resources/securing-infrastructure-in-application-pipelines) to validate your policies automatically during development and deployment. These tests ensure that only compliant policies are deployed to production.

Policy enforcement produces audit logs that track violations and generate compliance reports. This maintains visibility into infrastructure compliance status. Auditability is crucial for security and compliance, providing a clear record of policy violations and actions taken to address them.

## Use Sentinel with HCP Terraform

HCP Terraform integrates with [Sentinel](https://developer.hashicorp.com/terraform/enterprise/policy-enforcement/define-policies/custom-sentinel) to enforce policies during infrastructure provisioning and management. When a user runs `terraform plan` or `terraform apply`, Sentinel evaluates the defined policies against the proposed changes. When a policy is violated, Sentinel rejects the infrastructure changes that would be applied, and provides feedback to the user.

## Use OPA with Terraform

Open Policy Agent (OPA) is an open-source policy engine for defining and enforcing policies across your infrastructure. HCP Terraform integrates with [OPA](https://developer.hashicorp.com/terraform/enterprise/policy-enforcement/define-policies/opa) to execute policy checks before each plan or apply, preventing implementation of non-compliant changes. You can write OPA policies in Rego to validate resource configurations, enforce tagging standards, control costs, and ensure adherence to company or regulatory requirements. By integrating OPA with Terraform, you can automate policy enforcement and maintain compliance across your infrastructure.

## Manage cloud-native policies with Terraform

Many cloud providers offer native policy management tools, such as [AWS Config Rules](https://aws.amazon.com/blogs/mt/aws-organizations-aws-config-and-terraform/), [Enterprise Azure Policy as Code](https://azure.github.io/enterprise-azure-policy-as-code/), and [Google Cloud Organization Policy](https://cloud.google.com/blog/topics/developers-practitioners/implementing-iam-access-control-code-hashicorp-terraform). Terraform can manage these policies as code and integrate with these tools, ensuring cloud resources adhere to organizational standards and compliance requirements.

## Test policy as code

Policy as code is only as good as its tests. To ensure that your policies function as intended, you should implement a testing strategy for your policy code. This includes unit testing, integration testing, and continuous testing.

### Unit testing

Write unit tests for policy code to validate individual rules and logic using testing frameworks specific to your policy language (e.g., [Sentinel testing framework](https://developer.hashicorp.com/sentinel/docs/writing/testing)). These unit tests validate syntax, logic, and expected behavior during development.

### Integration testing

Test the integration of policies with infrastructure to verify enforcement during execution. This involves running tests that use the actual infrastructure to ensure that the policies work as expected. For example, if an application should have access to a specific secret, you can write an integration test to verify that the policy allows access to that secret while denying access to unauthorized resources.

### Continuous testing

Integrate policy tests into [CI/CD pipelines](https://developer.hashicorp.com/well-architected-framework/define-and-automate-processes/automate/cicd) to automatically run tests whenever changes occur to policy code. These tests might be unit or integration tests. They should run often during development and deployment, ensuring the deployment of compliant policies to production.
