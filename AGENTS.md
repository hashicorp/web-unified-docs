# AGENTS.md

This file provides guidance to agents when working with code in this repository.

## What this repo is

A Next.js app that acts as the unified docs API (`web-unified-docs`) serving all HashiCorp product documentation. It has two concerns: (1) a **prebuild pipeline** that processes `content/` into JSON manifests and transformed MDX files, and (2) **API route handlers** in `app/api/` that serve docs to `dev-portal`.

## Commands

```bash
npm test                  # run all vitest tests (watch mode)
npm run coverage          # run tests with v8 coverage report
npm run lint              # ESLint --fix on all JS/TS files
npm run format            # Prettier --write on all JS/TS files
npx vitest run <file>     # run a single test file (no watch)
npx vitest run -t "name"  # run a single test by name
npm run prebuild          # must run before `npm run build`; generates app/api/versionMetadata.json and app/api/docsPathsAllVersions.json
make                      # local preview via Docker Compose (dev-portal + unified-devdot-api)
```

> `npm run prebuild` must succeed before the app builds ŌĆö it generates `app/api/versionMetadata.json` and `app/api/docsPathsAllVersions.json`, which are imported at compile time.

## Prebuild binaries

`scripts/prebuild/run-prebuild.mjs` detects platform/arch and decompresses one of three pre-compiled Bun binaries (`prebuild-{arm-mac,arm-linux,x64-linux}-binary.gz`). The actual prebuild logic is in `scripts/prebuild/prebuild.mjs`.

## Path aliases (critical ŌĆö use these, not relative paths)

Defined in both `package.json#imports` and `tsconfig.json#paths`:

| Alias | Resolves to |
|---|---|
| `#utils/*` | `app/utils/*` |
| `#api/*` | `app/api/*` |
| `#productConfig.mjs` | `productConfig.mjs` |
| `#scriptUtils/*` | `scripts/utils/*` |

**Known Vitest quirk**: When mocking modules that use `#utils/*` aliases in the _subject under test_, the mock must use the same alias (`vi.mock('#utils/contentVersions', ...)`). But if the test _imports_ the subject with a relative path, use a relative path for the mock too ŌĆö Vitest doesn't always resolve aliases in `vi.mock()` correctly. See comment in [`app/utils/allDocsPaths.test.ts`](app/utils/allDocsPaths.test.ts:16).

## Error handling pattern

Use `Result<T, E>` from [`app/utils/result.ts`](app/utils/result.ts) (`Ok`/`Err`) instead of throwing exceptions. All API route handlers check `.ok` before accessing `.value`. Do not add `try/catch` where `Result` is already used.

## Code style (non-obvious rules)

- **No barrel files** ŌĆö `eslint-plugin-no-barrel-files` warns on index re-exports. Import directly from the module file.
- **No default exports** ŌĆö `no-restricted-exports` warns on `export default`. Use named exports everywhere.
- **Curly always** ŌĆö `curly: 'error'` means all `if`/`else`/`for` blocks require braces.
- **Arrow function bodies required** ŌĆö `'arrow-body-style': ['warn', 'always']` ŌĆö always use `() => { return x }` not `() => x`.
- Formatting: tabs (width 2), no semicolons, single quotes (enforced by Prettier + husky pre-commit).
- Every source file must start with the IBM copyright comment block.

## Testing patterns

- Tests live **next to** source files (e.g. `file.ts` ŌåÆ `file.test.ts`), not in a separate folder.
- `__fixtures__/` contains shared mock JSON and a fixture `productConfig.mjs` used by multiple tests.
- `__mocks__/fs.mjs` uses `memfs` ŌĆö when a test does `vi.mock('fs')` and `vi.mock('fs/promises')`, it picks up this in-memory FS automatically. Use `vol.reset()` in `beforeEach` to clear state.
- API route handlers are tested via [`app/utils/mockRequest.ts`](app/utils/mockRequest.ts) ŌĆö call `mockRequest(GET, { productSlug, version, docsPath })` instead of constructing `Request` objects manually.
- Use `vi.hoisted()` to set `process.env` values that are read at module load time (before imports).

## Content structure

- Versioned products: `content/<productSlug>/v1.x.x/` subdirectories.
- Non-versioned products: `content/<productSlug>/` flat (uses placeholder version `v0.0.x`).
- `productConfig.mjs` (`PRODUCT_CONFIG`) is the registry for all products ŌĆö add new products here.
- Do **not** manually edit `app/api/versionMetadata.json` or `app/api/docsPathsAllVersions.json` ŌĆö they are generated by prebuild.
- Do **not** edit content under `# START AUTO GENERATED METADATA, DO NOT EDIT` in MDX files.
